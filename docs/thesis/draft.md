# 一个轻量级 Python 虚拟机的设计与实现

# 摘要

# 第1章 绪论

## 1.1 课题背景

Python 语言凭借简洁的语法、较低的上手门槛以及丰富的第三方生态，已被广泛应用于 Web 开发、数据分析、人工智能、自动化运维等多个领域。作为一种动态类型高级语言，Python 程序通常并不直接运行在底层硬件之上，而是依赖 Python 虚拟机（Python Virtual Machine，PVM）完成编译、加载与执行。

虚拟机技术是现代高级编程语言的核心，它主要包括前端和后端两大部分。前端中由编译器（Compiler）将高级语言程序的源代码（Source Code）翻译成中间字节码（Bytecode）。后端中由解释器（Interpreter）负责执行字节码。此外，后端还提供垃圾回收器（Garbage Collector）和编程语言运行时（Runtime）等核心模块。这种机制使得 Python 或其他运用该项技术的高级编程语言，可以在任何安装了相应虚拟机软件的计算机系统上运行，而在源代码层面几乎不需要进行任何修改。

目前，最主流、最成熟的 Python 实现是官方维护的由 C 语言实现的 CPython。经过长期演进，CPython 在兼容性、稳定性和生态支持方面都已经非常完善。但从教学演示、轻量嵌入和定制化开发的角度来看，直接研究或裁剪一个大规模工业实现，往往存在两个现实困难。

一方面，CPython 作为长期维护的工业项目，且受限于 C 语言自身的表达与抽象能力，代码体量大、内部机制多，理解和修改的门槛较高。对于希望聚焦编程语言运行时核心机制的教学或实验场景而言，这种复杂度并不总是必要的。另一方面，许多嵌入式脚本场景更关心“够用、易嵌入、便于裁剪和二次开发”，而不是完整覆盖全部 Python 生态。在这些场景中，一个结构清晰、目标聚焦的轻量级 PVM 后端实现，具有一定的研究与工程价值。

基于上述背景，本文提出使用现代 C++20 语言设计并实现一个轻量级 Python 虚拟机后端系统，即 S.A.A.U.S.O VM。该系统以教学与嵌入场景为主要目标，兼容 CPython 3.12 字节码模型中的核心执行路径，支持 Python 语言的一个核心功能子集。本文中将重点讨论该系统中对象系统、解释器、异常机制、模块系统、内存管理以及嵌入接口等关键部分的设计思路与工程实现；相比直接讨论一个完整工业实现，围绕这些核心模块展开设计与实现，更有利于说明编程语言虚拟机的内部运行原理。

本文所实现的 S.A.A.U.S.O VM 已开源，相关链接见附录 A。

## 1.2 国内外研究现状

### 1.2.1 国外研究现状

在 Python 实现与编程语言虚拟机领域，国外已经积累了较为丰富的研究和工程成果，形成了多条有代表性的技术路线。

首先，在 Python 语言实现方面，CPython 仍然是事实上的标准实现，强调语言兼容性、稳定性和生态完整性。围绕 CPython 的替代或补充方案也持续出现。例如，PyPy 通过 JIT 技术提升运行效率，在动态语言性能优化方面具有代表性；MicroPython 面向资源受限设备，在体量和部署门槛方面具有明显优势；Codon 则尝试通过静态编译路径获得更高的执行效率。这些方案分别在兼容性、轻量化、运行效率和适用场景之间作出了不同取舍。

其次，从更广泛的编程语言虚拟机技术研究领域来看，Strongtalk、Java HotSpot 和 JavaScript V8 等工业级虚拟机的研究成果对后续语言运行时设计产生了深远影响。诸如对象模型分层、句柄系统、分代垃圾回收、内联缓存和运行时优化等思想，已经成为现代虚拟机设计中的重要参考。这些成果虽然并非直接面向 Python 语言，但它们在对象表示、执行模型和内存管理方面的设计经验，对构建新的 PVM 系统具有较强借鉴意义。

总体来看，国外相关研究已形成较为完整的生态：既有追求工业成熟度与语言完整性的实现，也有面向特定硬件平台或高性能需求的变体。这为本文选题提供了良好的比较对象和设计参考。

### 1.2.2 国内研究现状

相较于国外较成熟的虚拟机研究与开源生态，国内在 Python 语言实现和 PVM 方向上的公开研究相对较少，且整体更偏应用导向。

一类工作主要集中在 Python 的应用层使用，例如许多企业和高校会将 Python 用于 Web 服务、自动化脚本或数据处理任务，研究重点通常不在 PVM 本身。另一类工作则更多围绕 CPython 的扩展开发或源码剖析展开，这些工作对理解 Python 运行机制具有帮助，但通常并不以“从零设计并实现一个新的 PVM 后端”为目标。

在更广泛的编程语言虚拟机领域，国内部分互联网科技企业也会围绕具体业务场景定制专用虚拟机或运行时系统，例如字节跳动的 PrimJs、阿里巴巴的 DragonwellJDK 等。不过整体而言，这类成果往往偏工程内部使用，公开资料有限；结构清晰、适合教学或定制化修改的开源实现并不多见。

总体而言，国内在“轻量级、可嵌入、便于教学与定制化修改的 Python 虚拟机后端”这一交叉方向上，公开可参考的系统性实践仍然较少，这也进一步说明了本文工作的实际意义。

# 第2章 相关理论与关键技术基础

本章围绕 S.A.A.U.S.O VM 的实现目标，介绍轻量级 Python 虚拟机后端所涉及的理论基础与关键技术。这里讨论的重点并不是完整复述 Python 语言的所有语义细节，而是选取与本文后续系统设计和工程实现直接相关的部分，说明一个兼容 CPython 3.12 字节码模型的 PVM 后端通常需要解决哪些核心问题。

从整体上看，本文所实现的系统属于典型的“前端与后端分离”结构：前端负责生成或解析字节码形式的程序表示，后端负责字节码的装载、调度、执行以及运行时语义支撑。对后端而言，最关键的基础问题主要包括：如何组织字节码执行模型、如何管理函数调用与栈帧、如何表示 Python 对象及其类型关系、如何处理异常控制流，以及如何实现最小可用的模块加载机制。

这些内容既构成了 S.A.A.U.S.O VM 的理论基础，也直接决定了后续总体设计与关键模块实现的技术路线。

## 2.1 PVM 执行原理综述

Python 程序的执行通常分为两个阶段。第一阶段由编译器前端将源代码转换为字节码表示；第二阶段由 PVM 对字节码进行解释执行。对本文所讨论的系统而言，前端的具体实现方式并不是重点，关键在于后端如何接收一份已经编译好的代码对象，并据此完成运行时调度。

在 CPython 体系中，字节码程序通常由代码对象（CodeObject）描述。代码对象不仅保存字节码指令序列，还会记录符号名、常量表以及局部变量相关元信息。换句话说，代码对象相当于一段可执行程序的静态描述，而解释器真正运行时所需要的动态状态，则由后续的栈帧来承担。

从执行模型看，PVM 本质上是一种虚拟的堆栈机器（stack machine）。它在运行过程中维护操作数栈，并按照“取指、译码、执行”的顺序逐条处理字节码指令。许多字节码都会围绕栈顶元素展开：有的负责加载常量或变量，有的负责执行算术或逻辑运算，有的负责函数调用、属性访问或控制流跳转。当一条指令执行完成后，解释器再推进到下一条指令继续执行，直到当前代码对象的执行结束。

例如对于如下简单语句：

```python
x = 1
y = 2
z = 3
result = x + y * z
```

编译器会生成对应的字节码序列、符号表和常量表。解释器在执行时，会先将常量压入操作数栈，再通过存储类指令写入命名空间；当执行表达式 `x + y * z` 时，又会先将相关变量值取回栈顶，再按照字节码定义的顺序完成乘法和加法。对解释器而言，它并不直接“理解”高级语言语句，而是通过逐条执行字节码来间接实现高级语言语义。

需要指出的是，现代 PVM 的执行过程并不只是“有一个栈就够了”。为了支持函数调用、异常传播和模块加载，后端通常还需要维护程序计数器、当前栈帧、全局环境和异常状态等运行时信息。本文后续实现中采用的字节码解释器，正是在这种栈机模型的基础上，进一步结合代码对象、栈帧与运行时系统完成完整执行链路。

从工程实现角度看，这一节的重要意义在于说明：S.A.A.U.S.O VM 并不是直接执行 Python 源代码，而是围绕“代码对象 + 栈机解释器 + 运行时状态”这条主线展开设计。这也是后续解释器、对象系统和异常机制能够解耦实现的前提。

## 2.2 Python 中基本的控制流

控制流的本质，是决定程序下一步执行哪一条指令。与 CPU 硬件中的程序计数器类似，PVM 在解释执行过程中也需要维护一个程序计数器，用于标识当前或下一条待执行字节码的位置。顺序执行时，程序计数器按字节码顺序推进；而遇到条件分支、循环或异常等情况时，程序计数器则会被显式修改。

Python 中的 `if`、`while`、`break` 和 `continue` 等语句，在字节码层面都可以归结为若干条件跳转与无条件跳转指令的组合。也就是说，源代码层面看起来较为高级的控制结构，到了 PVM 内部，最终仍会被转化为对程序计数器的操作。

以 `while` 循环为例：

```python
while condition:
    i += 1
print(i)
```

编译后，解释器需要反复完成两件事：一是判断循环条件是否成立，二是在循环体执行完毕后重新回到条件检测位置。前者依赖条件跳转字节码，后者依赖后向跳转字节码。因此，从 PVM 的角度看，循环并不是一种独立机制，而是“条件判断 + 跳转”的组合。

在 Python 中，控制流判断还涉及一个容易被忽视的问题，即真值语义（truthiness）。许多语言只允许布尔值直接参与条件判断，但 Python 中的数字、字符串、容器乃至用户自定义对象，都可能被解释为真或假。这意味着解释器在执行条件跳转前，往往还需要借助运行时规则先将对象转换为逻辑上的真值结果。换句话说，控制流虽然表现为程序计数器的变化，但背后仍离不开对象系统与运行时语义的参与。

对本文后续实现而言，这一点非常关键。S.A.A.U.S.O VM 不仅需要支持基本跳转字节码，还必须让这些跳转能够和对象真值判定、异常传播以及函数调用过程协同工作。否则，即便单独实现了跳转指令，也无法真正还原 Python 程序中的控制流行为。

## 2.3 Python 中的函数

函数是 Python 程序中最重要的组织单位之一。对 PVM 而言，函数机制并不只是“执行一段可复用代码”这么简单，它还牵涉到代码对象创建、运行时函数对象构造、参数绑定、栈帧切换以及闭包变量访问等多个环节。对于一个可用的 Python 后端实现来说，函数机制几乎总是解释器设计中的核心部分。

### 2.3.1 简单函数的创建

在 Python 中，函数的静态部分和动态部分是分开的。静态部分对应编译阶段产生的代码对象，动态部分则是运行时创建的函数对象。也就是说，编译器首先会为每个函数体生成一个独立的代码对象；当程序真正运行到函数定义语句时，解释器再通过相应字节码把该代码对象包装为可调用的函数对象，并绑定到当前命名空间中。

例如：

```python
def add(a, b):
    return a + b

result = add(1, 2)
```

在这个过程中，`add` 函数对应的代码对象会先出现在外层代码对象的常量表中，随后由函数创建相关字节码在运行时构造出真正的函数对象。之所以要这样设计，是因为 Python 的函数本质上也是对象，它除了保存可执行代码外，还可能携带默认参数、闭包变量和所属全局环境等运行时信息。

这一区分对本文系统设计有直接影响。S.A.A.U.S.O VM 在后续实现中并不会把“函数”简单视为一段代码，而是将其建模为一种特殊对象：既包含代码对象，又持有与执行相关的上下文数据。这样才能进一步支持默认参数、闭包和方法绑定等更复杂的语义。

### 2.3.2 栈帧与函数调用的原理

每当 Python 程序发生一次函数调用，PVM 都需要为被调函数创建一个新的栈帧（Stack Frame）。栈帧用于保存该函数执行期间的动态状态，例如程序计数器、操作数栈、局部变量区以及与当前代码对象相关的上下文信息。

调用发生时，解释器会根据被调函数的代码对象和实参，建立新的栈帧，并将执行权切换到该栈帧。函数返回时，解释器再从当前栈帧中取出返回值，销毁或弹出该栈帧，并将控制权交还给调用方。这样，函数调用关系就表现为一组栈帧的入栈和出栈过程。

对于 PVM 而言，栈帧的意义远不止保存临时数据。它实际上承担了“局部执行上下文”的职责：不同函数拥有各自独立的局部变量、操作数栈和程序计数器，因此多个函数可以递归调用、相互调用或被异常中断，而不会彼此污染状态。本文后续解释器实现中的参数绑定、异常展开以及局部变量访问，都依赖于这一机制。

### 2.3.3 扩展参数与键值对参数

Python 函数调用语义比许多静态语言更灵活。除了普通的位置参数和关键字参数外，Python 还支持 `*args` 和 `**kwargs` 形式的扩展参数。前者用于接收多余的位置参数，后者用于接收未被显式形参消耗的键值对参数。

从 PVM 的角度看，这意味着“函数调用”并不仅仅是把一组实参按顺序塞进形参表中，而是需要经过一轮更复杂的参数归一化过程。解释器必须区分哪些实参属于普通位置参数，哪些属于关键字参数，哪些应被打包进 `tuple`，哪些应被收集进 `dict`，以及默认值在何时参与补全。

这一点对于实现来说非常重要。因为只要参数绑定规则稍有偏差，Python 函数调用的表现就会与预期不一致。本文所实现的 S.A.A.U.S.O VM 后端之所以专门将参数绑定作为解释器中的关键步骤之一，正是因为 Python 的函数语义本身具有较高复杂度，不能简单照搬静态语言函数调用模型。

### 2.3.4 函数闭包

闭包是 Python 词法作用域规则的重要体现。当内部函数引用其外层函数中的局部变量时，这些变量在外层函数返回后也不能立刻消失，而需要继续随闭包一起存活。为实现这一点，编译器会在编译阶段识别自由变量（free variables）和单元变量（cell variables），并生成相应的闭包访问字节码。

运行时，解释器需要把这些被捕获的变量包装为可共享的闭包单元，再在创建内部函数对象时将其与代码对象一并绑定。之后，内部函数执行时就可以通过相应字节码访问这些闭包变量，而不是从自己的局部变量区直接读取。

闭包的存在说明，函数对象并不只是一段“拿来执行的代码”，它还可能携带外部环境的一部分状态。对本文系统而言，这意味着函数对象模型必须考虑闭包支持，解释器在处理函数创建与调用时也必须兼顾自由变量访问路径。这部分内容会在后续函数对象与栈帧实现中进一步展开。

### 2.3.5 常用的内建函数（Builtin Function）

除了用户在源码中定义的函数之外，Python 运行时还提供一批内建函数，如 `print`、`len`、`isinstance` 等。这些函数在用户视角下与普通函数看起来相似，但它们通常并不是由 Python 源码直接定义，而是由虚拟机运行时预先注册到内建命名空间中。

对 PVM 而言，内建函数的重要性在于：它们是语言语义与运行时系统之间的重要桥梁。一方面，很多基础能力必须通过内建函数暴露给用户程序；另一方面，内建函数的调用过程又需要与普通函数调用协议相兼容，保证从解释器视角看它们都能被统一调度。

因此，在一个轻量级 PVM 后端中，内建函数系统通常既是语言可用性的基础，也是解释器与运行时协作的关键接口。本文后续实现中同样采用了这一思路：将内建函数作为运行时预置能力的一部分进行统一管理。

## 2.4 Python 中的面向对象机制

Python 是一门面向对象的动态语言。对象系统不仅影响类和实例本身，还会影响属性访问、方法调用、内建类型表示、继承与多态等一系列运行时行为。对于 PVM 实现来说，对象系统往往是贯穿解释器、运行时和内存管理的核心基础设施。

### 2.4.1 综述

在 Python 中，万物皆对象。数字、字符串、列表、函数、类乃至模块，都可以统一看作 Python 对象。与此同时，Python 还将“类型”本身也对象化，即类同样可以作为对象被传递和操作。这种设计使得 Python 具有较强的灵活性，但也意味着 PVM 需要建立一套统一的对象表示机制，来同时描述“对象实例”和“对象类型”。

对于本文系统而言，对象模型并不只是语法层面的“类和对象”概念，而是整个运行时的统一数据组织方式。解释器执行过程中操作的栈值、运行时中传递的函数对象、模块系统中维护的模块对象，最终都要落在统一的对象表示之上。

### 2.4.2 基本内建类型

一个最小可用的 PVM 后端，通常需要先支持若干基础内建类型，才能承载更高层语言语义。对 Python 而言，这些类型至少包括 `object`、`int`、`float`、`str`、`list`、`dict` 等。它们分别承担了不同角色：`object` 为用户自定义类提供最基础的祖先类型；数值与字符串类型支撑最常见的数据表达；列表和字典则是控制流、函数调用和模块机制中频繁使用的核心容器。

这些内建类型并不是简单的数据结构集合。它们还携带各自的运行时行为，例如字符串连接、列表下标访问、字典查找、对象比较和真值判定等。因此，在 PVM 中实现内建类型，既要考虑对象布局，也要考虑相关运算和方法语义如何被统一调度。

### 2.4.3 类的封装

封装的核心含义，是将对象状态与围绕该状态运行的方法组织在一起。Python 同样支持这一基本思想：用户可以在类中定义属性和方法，再由实例对象持有具体状态，通过方法对外提供行为。

不过，Python 的封装方式与 C++、Java 等语言并不完全相同。它并不强调严格的访问控制修饰符，而更强调运行时动态性和约定式使用方式。这意味着 PVM 在实现对象系统时，不能简单照搬静态语言的类模型，而是需要让对象属性、实例字典、类属性查找以及方法绑定机制能够相互协作。

对本文系统来说，这一点直接影响对象模型设计：对象实例需要持有自身状态，类型对象需要持有方法和属性定义，而属性访问过程还要能够在实例与类型之间完成查找与分派。

### 2.4.4 类的继承

继承允许一个类在已有类的基础上扩展行为和结构，是 Python 面向对象机制中的重要组成部分。通过继承，子类可以复用父类的属性和方法，也可以重写父类方法形成新的行为。

除了单继承之外，Python 还支持多继承。多继承带来的核心问题是：当多个父类中存在同名属性或方法时，解释器应当按照什么顺序进行查找。为此，Python 引入了方法解析顺序（Method Resolution Order，MRO）的概念，并采用 C3 线性化算法生成类层次结构中的查找顺序。

从 PVM 实现角度看，继承并不仅仅意味着“复制父类方法”，更关键的是建立一条稳定、可计算的查找路径。后续对象属性访问、方法绑定和内建运算分派，都可能依赖这条路径。因此，一个支持 Python 对象模型的后端，必须将类型关系、继承结构和 MRO 查找机制纳入对象系统设计之中。

### 2.4.5 类的多态

多态指相同的调用接口在不同对象上表现出不同的行为。对于 Python 来说，多态并不只来源于传统的继承和方法重写，也和动态类型语言常见的鸭子类型特征密切相关。只要一个对象提供了调用方所需的方法或协议，它就可以在运行时被接受。

这对 PVM 的启发在于：运行时系统不能过早地把对象行为写死在某一种静态类型上，而是需要依赖统一的属性查找、方法分派或协议调用机制，在运行阶段根据对象实际类型决定行为。也正因如此，Python 对象系统的实现通常会把“对象表示”和“行为调度”区分开，使得多态分派能够在更统一的入口上完成。

对本文后续实现而言，多态不仅体现在用户自定义类的方法查找上，也体现在内建类型协议、运算语义和内建函数调用等多个方面。因此，本节讨论的对象模型，并不是局部特性，而是整个运行时语义的基础。

## 2.5 Python 中的异常机制

异常机制本质上是一种特殊的控制流机制。与普通的顺序执行或条件跳转不同，异常会在运行时打断当前指令流，并尝试将控制权转移到最近的异常处理器处。因此，对 PVM 而言，异常机制并不是附属功能，而是解释器主执行路径必须原生支持的一部分。

Python 的异常处理以异常对象为中心。当程序执行过程中发生错误，或用户显式执行 `raise` 语句时，解释器会创建或传播一个异常对象，并记录当前的异常状态。之后，运行时需要判断当前执行位置是否存在可匹配的 `except` 或 `finally` 处理器；如果存在，则转入相应处理器继续执行；如果不存在，则继续向调用栈更外层展开。

自 CPython 3.11 起，异常处理的一个重要特征是引入函数级异常表（exception table）。也就是说，解释器不再单纯依赖早期那种显式维护块栈的方式来描述异常处理范围，而是在异常发生后，根据当前代码对象携带的异常表信息定位处理器，并完成异常展开。这种方式使得正常执行路径上的异常处理开销更低，同时也让异常控制流与普通控制流之间的分工更清晰。

对一个兼容现代 CPython 字节码模型的 PVM 后端而言，异常机制至少要解决四个问题：如何表示异常状态，如何在抛出后延续传播，如何根据异常表查找处理器，以及如何在跨函数调用时进行栈帧级展开。本文后续实现中，异常系统正是围绕这几个问题展开设计的，因此本节内容实际上为解释器与运行时系统的后续实现提供了理论基础。

## 2.6 Python 中的模块机制

模块机制是 Python 组织代码和复用功能的重要方式。对用户而言，`import` 语句看起来很简单；但对 PVM 来说，模块导入涉及名称解析、模块查找、代码加载、模块体执行、导入缓存以及父子模块绑定等多个环节，是一个典型的系统级功能。

从概念上看，模块既是一段代码的载体，也是一层命名空间边界。解释器执行 `IMPORT_NAME`、`IMPORT_FROM` 等相关字节码时，运行时通常需要先判断目标模块是否已存在于缓存中；若不存在，则按照搜索路径查找模块文件或包入口，创建模块对象，并执行模块体初始化其命名空间。执行完成后，该模块会被登记到 `sys.modules`，供后续重复导入时复用。

对于包（package）来说，模块机制还需要额外处理父子层级关系、相对导入级别（level）以及 `from ... import ...` 的绑定语义。这也是为什么模块系统在实现上往往不能只写成一个简单的“打开文件并执行”过程，而需要拆分出查找、解析、加载和缓存等多个职责明确的阶段。

对本文而言，模块系统之所以值得在理论基础中单独讨论，是因为 S.A.A.U.S.O VM 并不只是要运行一段孤立脚本，而是需要支持更接近真实 Python 程序结构的代码组织方式。只有具备最小可用的模块机制，解释器、运行时和嵌入接口才能真正形成一个更完整的系统闭环。
