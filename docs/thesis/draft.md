# 一个轻量级 Python 虚拟机的设计与实现

# 第1章 绪论

## 1.1 课题背景

Python 语言凭借简洁的语法、较低的上手门槛以及丰富的第三方生态，已被广泛应用于 Web 开发、数据分析、人工智能、自动化运维等多个领域。作为一种动态类型高级语言，Python 程序通常并不直接运行在底层硬件之上，而是依赖 Python 虚拟机（Python virtual machine，PVM）完成编译、加载与执行。

虚拟机技术是现代高级编程语言的核心，它主要包括前端和后端两大部分。前端中由编译器（compiler）将高级语言程序的源代码（source code）翻译成中间字节码（bytecode）。后端中由解释器（interpreter）负责执行字节码。此外，后端还提供垃圾回收器（garbage collector）和编程语言运行时（runtime）等核心模块。这种机制使得 Python 或其他运用该项技术的高级编程语言，可以在任何安装了相应虚拟机软件的计算机系统上运行，而在源代码层面几乎不需要进行任何修改。

目前，最主流、最成熟的 Python 实现是官方维护的由 C 语言实现的 CPython。经过长期演进，CPython 在兼容性、稳定性和生态支持方面都已经非常完善。但从教学演示、轻量嵌入和定制化开发的角度来看，直接研究或裁剪一个大规模工业实现，往往存在两个现实困难。

一方面，CPython 作为长期维护的工业项目，且受限于 C 语言自身的表达与抽象能力，代码体量大、内部机制多，理解和修改的门槛较高。对于希望聚焦编程语言运行时核心机制的教学或实验场景而言，这种复杂度并不总是必要的。另一方面，许多嵌入式脚本场景更关心“够用、易嵌入、便于裁剪和二次开发”，而不是完整覆盖全部 Python 生态。在这些场景中，一个结构清晰、目标聚焦的轻量级 PVM 后端 *（为行文方便，如无特殊说明，后文中出现的 PVM 均指 PVM 后端）* 实现，具有一定的研究与工程价值。

基于上述背景，本文提出使用现代 C++20 语言设计并实现一个轻量级 Python 虚拟机后端系统，即 S.A.A.U.S.O VM。该系统以教学与嵌入场景为主要目标，兼容 CPython 3.12 字节码模型，并支持 Python 语言的一个核心功能子集。本文中将重点讨论该系统中对象系统、解释器、异常机制、模块系统、内存管理以及嵌入接口等关键部分的设计思路与工程实现；相比直接讨论一个完整工业实现，围绕这些核心模块展开设计与实现，更有利于说明编程语言虚拟机的内部运行原理。

本文所实现的 S.A.A.U.S.O VM 已开源，相关链接见附录 A。

## 1.2 国内外研究现状

### 1.2.1 国外研究现状

在 Python 实现与编程语言虚拟机领域，国外已经积累了较为丰富的研究和工程成果，形成了多条有代表性的技术路线。

首先，在 Python 语言实现方面，CPython 仍然是事实上的标准实现，强调语言兼容性、稳定性和生态完整性。围绕 CPython 的替代或补充方案也持续出现。例如，PyPy 通过 JIT 技术提升运行效率，在动态语言性能优化方面具有代表性；MicroPython 面向资源受限设备，在体量和部署门槛方面具有明显优势；Codon 则尝试通过静态编译路径获得更高的执行效率。这些方案分别在兼容性、轻量化、运行效率和适用场景之间作出了不同取舍。

其次，从更广泛的编程语言虚拟机技术研究领域来看，Strongtalk、Java HotSpot 和 JavaScript V8 等工业级虚拟机的研究成果对后续语言运行时设计产生了深远影响。诸如对象模型分层、句柄系统、分代垃圾回收、内联缓存和运行时优化等思想，已经成为现代虚拟机设计中的重要参考。这些成果虽然并非直接面向 Python 语言，但它们在对象表示、执行模型和内存管理方面的设计经验，对构建新的 PVM 系统具有较强借鉴意义。

总体来看，国外相关研究已形成较为完整的生态：既有追求工业成熟度与语言完整性的实现，也有面向特定硬件平台或高性能需求的变体。这为本文选题提供了良好的比较对象和设计参考。

### 1.2.2 国内研究现状

相较于国外较成熟的虚拟机研究与开源生态，国内在 Python 语言实现和 PVM 方向上的公开研究相对较少，且整体更偏应用导向。

一类工作主要集中在 Python 的应用层使用，例如许多企业和高校会将 Python 用于 Web 服务、自动化脚本或数据处理任务，研究重点通常不在 PVM 本身。另一类工作则更多围绕 CPython 的扩展开发或源码剖析展开，这些工作对理解 Python 运行机制具有帮助，但通常并不以"从零设计并实现一个新的 PVM"为目标。

在更广泛的编程语言虚拟机领域，国内部分互联网科技企业也会围绕具体业务场景定制专用虚拟机或运行时系统，例如字节跳动的 PrimJs、阿里巴巴的 DragonwellJDK 等。不过整体而言，这类成果往往偏工程内部使用，公开资料有限；结构清晰、适合教学或定制化修改的开源实现并不多见。

总体而言，国内在“轻量级、可嵌入、便于教学与定制化修改的 Python 虚拟机后端”这一交叉方向上，公开可参考的系统性实践仍然较少，这也进一步说明了本文工作的实际意义。

## 1.3 课题研究的内容及意义

综合来看，现有 Python 实现方案往往各有侧重：有的强调完整兼容性，有的强调运行效率，有的强调在极小硬件上的可部署性，但在“轻量化、便于嵌入、结构清晰、相对易于教学演示和二次开发”这几个维度上，同时兼顾的公开实现并不多见。

- 设计并实现一个基于 C++20 的轻量级 PVM，支持 Python 语言的核心功能子集，并兼容 CPython 3.12 字节码执行模型中的关键路径；

### （3）人才培养价值

# 第2章 PVM 理论基础与关键技术

本章将围绕 S.A.A.U.S.O VM 的实现目标，介绍轻量级 PVM 所涉及的理论基础与关键技术。

本章中并不会完整复述 Python 语言的所有功能细节，而是围绕 S.A.A.U.S.O VM 所支持的 Python 语言核心功能，分析并说明一个兼容 CPython 3.12 字节码模型的轻量级 PVM 通常需要具备哪些关键能力，以及需要解决哪些关键问题。

## 2.1 PVM 执行原理概述

### 2.1.1 CPython 执行模型概述

Python 程序的执行通常分为两个阶段。第一阶段由编译器前端将源代码翻译为字节码表示；第二阶段由虚拟机后端对字节码进行解释执行。由于本文主要关注 PVM 后端系统的设计与实现，因此不会对前端的具体实现方式做过多探讨，而是重点聚焦 PVM 后端如何接收一份已经翻译好的字节码表示，并据此完成运行时行为。

在 CPython 的执行模型中，字节码程序通常由代码对象（code object）描述。代码对象不仅保存字节码指令序列，还会记录符号名列表、常量列表等元信息。换句话说，代码对象相当于一段 Python 程序的静态描述；至于 Python 程序运行过程中的动态状态，则在后文中将要探讨的 PVM 栈帧（stack frame）中进行维护。

从执行模型看，PVM 本质上是一种虚拟的栈机（stack machine）。它在运行过程中维护操作数栈，并按照“取指、译码、执行”的顺序逐条处理字节码指令；当一条指令执行完成后，PVM 再根据指令计数器（program counter）推进到下一条指令继续执行，直到当前代码对象的执行结束。

在 CPython 的执行模型中，每条字节码指令由操作码（operation code）和操作参数（operation parameter）两部分构成，每个部分各需 1 个字节进行表示。字节码指令是 CPython 执行模型的核心，它们的职责覆盖常量加载、变量读写、算术或逻辑运算、函数调用、属性访问和控制流跳转等核心语义，它们经过编译器前端的排列组合，从而得以表达一段 Python 程序的完整动作流程。

例如对于如下简单语句：

```python
x = 1
y = 2
z = 3
x + y * z
```

编译器前端生成的代码对象，核心内容如下：
```
【字节码指令序列】
地址       操作码                  参数
  2      LOAD_CONST               0 (1)
  4      STORE_NAME               0 (x)

  6      LOAD_CONST               1 (2)
  8      STORE_NAME               1 (y)

 10      LOAD_CONST               2 (3)
 12      STORE_NAME               2 (z)

 14      LOAD_NAME                0 (x)
 16      LOAD_NAME                1 (y)
 18      LOAD_NAME                2 (z)
 20      BINARY_OP                5 (*)
 24      BINARY_OP                0 (+)

【常量列表】
[1, 2, 3]

【符号名列表】
['x', 'y', 'z']
```

PVM 在执行时，会先通过 `LOAD_CONST` 指令从常量列表中读取元素并压入操作数栈，再通过 `STORE_NAME` 指令将操作数栈顶的元素弹出并存储为全局变量。当执行表达式 `x + y * z` 时，又会先通过 `LOAD_NAME` 指令读取相关全局变量值压入栈顶，再通过 `BINARY_OP` 指令消费栈顶元素并完成乘法和加法。由此可见，对 PVM 而言，它并不直接"理解" Python 这一高级语言本身，而是通过逐条解释执行字节码来间接实现高级语言的语义。

需要指出的是，实际 PVM 的执行过程并不只是维护一个操作数栈那么简单。为了支持函数调用、异常传播、模块加载以及各种更加复杂的语言功能，PVM 通常还需要维护函数调用堆栈、全局环境（global）和异常状态（exception state）等状态信息，并且依赖语言运行时子系统（runtime）提供服务。

一般地，在较为复杂的工业级 VM 系统中，会将负责逐条解释执行字节码指令、维护指令计数器和函数调用堆栈的核心逻辑部分抽象成一个子系统，称为解释器（interpreter），本文中实现的 S.A.A.U.S.O VM 同样借鉴了这一思路。本文后续将继续探讨，如何在基于栈机模型的解释器的基础上，进一步结合对象模型、模块、运行时等子系统，构建可实际运行的轻量级 PVM 系统。

### 2.1.2 Python 中的命名空间与变量绑定机制概述

在 Python 中，变量并不完全等价于 C/C++ 等语言中某个固定内存地址上的“存储单元”，而更接近于一种“名称到对象的绑定关系”。在 Python 程序执行过程中，变量读取操作在本质上需要 PVM 将一个变量名解析到其所对应的对象上；赋值操作在本质上，也是在某个命名空间中建立或更新这种绑定关系。

从 PVM 的角度看，Python 至少涉及几类最常见的命名空间。首先是函数调用过程中形成的局部命名空间（local namespace），它通常用于保存形参与局部变量；其次是模块或脚本顶层对应的全局命名空间（global namespace），它承担了顶层变量、顶层函数和类定义的保存职责；此外，PVM 还维护内建命名空间（builtin namespace），用于提供内建函数。对于闭包场景，还需要通过额外的闭包单元（closure cell）来维持自由变量（free variable）的跨栈帧可见性，这在后文中会详细探讨。

例如，在下面的例子中，`add` 函数和变量 `c` 属于全局命名空间，`add` 函数中的形参 `a`、`b` 和局部变量 `result` 属于调用该函数而产生的局部命名空间，`print` 函数属于内建命名空间。

```python
def add(a, b):
  result = a + b
  return result

c = add(1, 2)
print(c)
```

这意味着，PVM 在实现变量读写时，不能简单类比 C/C++ 等语言，把“读取变量”和“写入变量”理解为对单一内存位置的直接访问，而必须结合当前局部命名空间、全局命名空间、内建命名空间和闭包环境共同完成名字解析。局部变量、全局变量和自由变量在 PVM 中往往具有不同的查找逻辑，也会对应不同类别的字节码访问方式（如 `LOAD_FAST`、`LOAD_NAME`、`LOAD_DEREF`）。

对本文后续 PVM 的实现而言，这一机制具有直接影响。其一，S.A.A.U.S.O VM 中的栈帧需要维护局部执行环境；其二，顶层脚本需要提供全局命名空间；其三，闭包函数需要额外保存被捕获的自由变量。换言之，变量机制并不是 PVM 中的一个局部语法问题，而是贯穿代码对象、栈帧、函数对象和运行时行为的共同基础。

## 2.2 Python 中基本的控制流

控制流的本质，是决定程序下一步执行哪一条指令。与 CPU 硬件中的程序计数器类似，PVM 在解释执行过程中也需要维护一个程序计数器，用于标识下一条待执行字节码的位置。顺序执行时，程序计数器按字节码顺序推进；而遇到条件分支、循环或异常等情况时，程序计数器则会被跳转相关的字节码显式修改。

Python 中的 `if`、`while`、`break` 和 `continue` 等语句，在字节码层面都可以归结为若干条件跳转与无条件跳转指令的组合。也就是说，源代码层面看起来较为高级的控制结构，到了 PVM 内部，实际会被转化为对程序计数器的操作。

以 `while` 循环为例：

```python
while condition:
    i += 1
print(i)
```

编译器前端生成的字节码指令序列，核心内容如下：
```
地址       操作码                      参数
  2      LOAD_NAME                0 (condition)
  4      POP_JUMP_IF_FALSE        6 (to 18)

  6      LOAD_NAME                1 (i)
  8      LOAD_CONST               0 (1)
 10      BINARY_OP               13 (+=)
 14      STORE_NAME               1 (i)

 16      JUMP_BACKWARD            8 (to 2)

 18      后续代码略...
```

分析编译结果可知，在运行时这段代码时 PVM 需要反复完成两件事：一是判断循环条件是否成立，二是在循环体执行完毕后重新回到条件检测位置。与汇编程序类似，前者依赖条件跳转字节码（如 `POP_JUMP_IF_FALSE`、`POP_JUMP_IF_TRUE` 等），后者依赖后向跳转字节码（如 `JUMP_BACKWARD`）。因此，从 PVM 的角度看，循环并不是一种独立机制，而是“条件判断 + 跳转”的组合。

在 Python 中，控制流判断还涉及一个容易被忽视的问题，即真值语义（truthiness）。在 Java 等语言中，只允许布尔值直接参与条件判断；但 Python 中的数字、字符串、容器乃至用户自定义对象，都允许被 PVM 解释为真或假。这意味着 PVM 在执行条件跳转前，往往还需要借助运行时规则先将对象转换为逻辑上的真值结果。换句话说，控制流虽然表现为程序计数器的变化，但背后仍离不开对象系统与运行时语义的参与。

对本文后续 PVM 的实现而言，这一点非常关键。S.A.A.U.S.O VM 不仅需要支持基本跳转字节码，还必须让这些跳转能够和对象真值判定等运行时能力协同工作。否则，即便单独实现了跳转指令，也无法真正实现 Python 程序中的控制流行为。

## 2.3 Python 中的函数

函数是 Python 程序中最重要的组织单位之一。对 PVM 而言，函数机制并不只是“执行一段可复用代码”这么简单，它还牵涉到代码对象创建、运行时函数对象构造、参数绑定、栈帧切换以及闭包变量访问等多个环节。

### 2.3.1 简单函数的创建

不同于 C/C++ 等静态语言，在 CPython 的执行模型中，函数的静态部分和动态部分是分开的。静态部分对应编译阶段产生的代码对象，动态部分则是运行阶段由 PVM 动态创建的函数对象。也就是说，编译器首先会为每个函数体生成一个独立的代码对象；当程序真正运行到函数定义语句时，解释器再通过 `MAKE_FUNCTION` 字节码把该代码对象包装为可调用的函数对象，并绑定到当前命名空间中。

例如：

```python
def add(a, b):
    return a + b

result = add(1, 2)
```

编译器前端生成的字节码指令序列中，创建 `add` 函数对象的过程如下：
```
地址       操作码                      参数
 2      LOAD_CONST               0 (<code object add at 0x5b596d3b75e0>)
 4      MAKE_FUNCTION            0
 6      STORE_NAME               0 (add)
```

分析编译结果可知，在这个过程中，`add` 函数对应的代码对象会先出现在外层代码对象的常量列表中，随后由 `MAKE_FUNCTION` 字节码在运行时构造出真正的函数对象。

下面再通过两个例子进一步说明 CPython 执行模型中采用这种动态创建函数对象机制的必要性。

在 Python 中，允许为函数的形参设置默认值。并且不同于 C++ 等静态语言，Python 中的函数参数默认值，可以是一个运行时阶段动态创建的变量。例如：
```python
default_k = 3
def linear_f(x, b, k = default_k):
    return k * x + b
```

这个过程是如何实现的呢？分析编译器前端生成的字节码指令序列，不难找到答案：
```
地址       操作码                      参数
  6      LOAD_NAME                0 (default_k)
  8      BUILD_TUPLE              1
 10      LOAD_CONST               1 (<code object linear_f at 0x57ba48f76b90>)
 12      MAKE_FUNCTION            1 (defaults)
 14      STORE_NAME               1 (linear_f)
```

从这段指令序列中，我们了解到 Python 中创建带有默认值参数的函数时，首先会执行 `BUILD_TUPLE` 字节码动态创建一个打包所有参数默认值的 tuple 容器对象并压回栈顶。随后在创建函数对象时，`MAKE_FUNCTION` 字节码会一并消费栈顶的该 tuple 容器对象和静态对象，将它们绑定进最终得到的函数对象当中。这个例子说明，Python 中为了实现函数对象与动态创建的参数默认值的绑定，需要函数对象在运行时阶段被动态地创建。

另一个例子与 Python 语言的模块系统有关。假设有两个模块 `mod_a.py` 和 `mod_b.py`：
```python
# mod_a.py
x = 10
def func():
    print(x)
```

```python
# mod_b.py
from mod_a import func
x = 999
func()
```

当我们执行 `mod_b.py` 后，得到的输出结果为 10 而非 999。这意味着当 `mod_a.py` 中的 func 函数被创建时，它会与所在 Python 模块的全局命名空间发生动态绑定。这个例子说明，Python 中为了实现函数对象与所在模块的全局命名空间绑定，需要函数对象在运行时阶段被动态地创建。

到此为止，我们已经基本明确 Python 中函数对象的动态创建机制，以及需要这种动态机制的原因。这对本文后续的 PVM 系统设计有直接影响。S.A.A.U.S.O VM 在后续实现中不应当把“函数”简单视为一段代码，而是将其建模为一种特殊对象：既包含代码对象等静态的元数据，又持有运行时阶段产生的上下文信息。这样才能进一步支持默认参数等更复杂的语义。

### 2.3.2 栈帧与函数调用的原理

Python 中的函数调用是通过 `CALL` 等字节码

类似于大多数编程语言，每当 Python 程序发生一次函数调用，PVM 都需要为被调函数创建一个新的栈帧。栈帧用于保存该函数执行期间的动态状态，例如程序计数器、操作数栈、局部变量区以及与当前被调函数相关的上下文信息。

调用发生时，PVM 会根据被调函数的代码对象和实参，建立新的栈帧，并将执行权切换到该栈帧。函数返回时，PVM 再从当前栈帧中取出返回值，销毁或弹出该栈帧，并将控制权交还给调用方。这样，函数调用关系就表现为一组栈帧的入栈和出栈过程。

对于 PVM 而言，栈帧的意义远不止保存临时数据。它实际上承担了“局部执行上下文”的职责：不同函数拥有各自独立的局部变量区、操作数栈和程序计数器，从而才使得函数之间可以相互调用，以及确保被调函数发生异常中断时，不会污染调用方函数的状态。本文后续 PVM 实现中的局部变量访问、栈展开等功能，都依赖于这一机制。

对于栈帧的局部变量区，这里可以作进一步的探讨。在 CPython 的执行模型中，站在 PVM 的视角，栈帧的局部变量区可以被视作一个定长数组，能够通过 `LOAD_FAST`、`STORE_FAST` 等字节码指令直接按槽位下标进行读写，而不需要像通过全局变量那样进行开销昂贵的查找操作。这种设计消除了读写局部变量时解析变量名所带来的运行时开销。

### 2.3.3 关键字参数、扩展参数与键值对参数

Python 函数调用语义比许多静态语言更灵活。除了普通的位置参数外，Python 还支持关键字参数以及 `*args` 和 `**kwargs` 形式的扩展参数。关键字参数用于对指定名称的形参进行赋值；`*args` 代表一个 `tuple` 容器，用于接收多余的位置参数；`**kwargs` 代表一个 `dict` 容器，用于接收未被显式形参消耗的键值对参数。

从 PVM 的角度看，这意味着“函数调用”并不仅仅是简单地把一组实参按顺序打包进形参表中，而是需要经过一轮更复杂的参数归一化过程。PVM 必须区分哪些实参属于普通位置参数，哪些属于关键字参数，哪些应被打包进 `tuple`，哪些应被收集进 `dict`，以及默认值在何时参与补全。

这一点对于实现来说非常重要。因为只要参数绑定规则稍有偏差，Python 函数调用的表现就会与预期不一致。考虑到 Python 的函数语义本身具有较高复杂度，不能简单照搬静态语言函数调用模型，本文所实现的 S.A.A.U.S.O VM 选择将参数绑定逻辑提取为虚拟机中的解释器下属的专门模块，以便独立进行维护。

### 2.3.4 函数闭包

不同于 C/C++，Python 中支持嵌套定义函数，因此进一步会延伸出闭包（closure）机制。闭包是 Python 词法作用域规则的重要体现。

首先通过一个例子，简要说明 Python 语言中闭包的概念和特点：
```python
def outer():
    x = 10
    def inner():
        print(x)
    return inner
```

在这段代码中，内部函数 `inner` 对外部函数 `outer` 中的局部变量 `x` 形成引用，因此 `inner` 被称为闭包函数（closure function），变量 `x` 则被称为 `inner` 的自由变量（free variable）。

站在 PVM 的视角，要实现闭包机制，需要解决几个核心问题。其一，PVM 需要能够区分普通的函数局部变量和自由变量。其二，PVM 需要保证外部函数和内部闭包函数均支持对自由变量进行读写。其三，PVM 需要保证闭包函数中持有的是对自由变量的真实引用，而非副本。其四，PVM 需要保证自由变量在外部函数返回后不会立即消失，而是继续随内部函数一起存活。

下面给出编译器前端生成的字节码指令序列的核心内容，用于进一步说明 CPython 执行模型中闭包的实现机制：

```
<code object outer>:
地址       操作码                      参数
  0      MAKE_CELL                1 (x)

  2      RESUME                   0

  4      LOAD_CONST               1 (10)
  6      STORE_DEREF              1 (x)

  8      LOAD_CLOSURE             1 (x)
 10      BUILD_TUPLE              1
 12      LOAD_CONST               2 (<code object inner>)
 14      MAKE_FUNCTION            8 (closure)
 16      STORE_FAST               0 (inner)

 18      LOAD_FAST                0 (inner)
 20      RETURN_VALUE

<code object inner>:
地址       操作码                      参数
 0      COPY_FREE_VARS           1

 2      RESUME                   0

 4      LOAD_GLOBAL              1 (NULL + print)
14      LOAD_DEREF               0 (x)
16      CALL                     1
24      POP_TOP
26      RETURN_CONST             0 (None)
```

首先在编译阶段，编译器负责识别自由变量，并生成不同于普通局部变量的读写字节码，如 `LOAD_DEREF`、`STORE_DEREF` 等。

运行时，解释器通过 `MAKE_CELL` 创建闭包单元（cell）来承载对自由变量值的引用，这为将对自由变量的引用传递给内部函数提供了媒介；，再在创建内部函数对象时将其与代码对象一并绑定。之后，内部函数执行时就可以通过相应字节码访问这些闭包变量，而不是从自己的局部变量区直接读取。

闭包的存在说明，函数对象并不只是一段“拿来执行的代码”，它还可能携带外部环境的一部分状态。对本文系统而言，这意味着函数对象模型必须考虑闭包支持，解释器在处理函数创建与调用时也必须兼顾自由变量访问路径。这部分内容会在后续函数对象与栈帧实现中进一步展开。

### 2.3.5 内建函数

除了用户在源码中定义的函数之外，PVM 还负责提供一批内建函数（builtin function），如 `print`、`len`、`isinstance` 等。这些函数在用户视角下与普通函数看起来相似，但它们通常并不是由 Python 源码直接定义，而是由虚拟机自身的 native 开发语言直接实现在 VM 内部，并预先注册到 Python 程序的内建命名空间中的。

类似于操作系统内核提供的系统调用，对 PVM 而言，内建函数的重要性在于它们是语言语义与虚拟机系统之间的重要桥梁，很多基础能力必须通过内建函数才能暴露给用户程序。

鉴于内建函数的重要性，在本文所实现的 S.A.A.U.S.O VM 中，选择将具体内建函数的实现作为虚拟机执行层的重要组成部分，进行统一管理。

另外需要指出的是，在 PVM 中，实现内建函数并非简单编写几个 native 函数那么简单。首先，从编译器前端的视角来看，它并不清楚一个被用户调用的函数，究竟是一个内建函数，还是一个普通的 Python 函数。因此编译器前端只会生成通用的函数调用字节码。其次，从用户程序的视角来看，无论是内建函数还是普通的 Python 函数，它们都是一个可调用 Python 对象。

因此为了抹平这种差异，在本文所实现的 S.A.A.U.S.O VM 中，选择将普通函数和内建函数采用统一的 Python 对象形态进行表示；在虚拟机系统实际执行函数调用操作时，才予以区分并执行不同的实际调度操作。

## 2.4 Python 中的面向对象机制

Python 是一门面向对象的动态语言。对象系统不仅影响类和实例本身，还会影响属性访问、方法调用、内建类型表示、继承与多态等一系列运行时行为。对于 PVM 实现来说，对象系统往往是贯穿解释器、运行时和内存管理的核心基础设施。

### 2.4.1 综述

在 Python 中，万物皆对象。数字、字符串、列表、函数、类乃至模块，都可以统一看作 Python 对象。与此同时，不同于 C++，Python 还将“类型”本身也对象化，即类同样可以作为对象被传递和操作。这种设计使得 Python 具有较强的灵活性，但也意味着 PVM 需要建立一套统一的对象表示机制，来同时描述“对象实例”和“对象类型”。

对于 PVM 而言，对象模型并不只是语法层面的“类和对象”概念，而是整个运行时的统一数据组织方式。在 PVM 内部，解释器执行过程中操作的栈值、运行时中传递的函数对象、模块系统中维护的模块对象，最终都要落在统一的对象表示之上。

### 2.4.2 基本内建类型

一个最小可用的 PVM 后端，通常需要先支持若干基础内建类型，才能承载更高层语言语义。对 Python 而言，这些类型至少包括 `object`、`int`、`float`、`str`、`tuple`、`list`、`dict` 等。它们分别承担了不同角色：`object` 是一切 Python 类型的祖先类型；数值与字符串类型支撑最常见的数据表达；元组、列表和字典则是函数创建、函数调用和模块机制等虚拟机核心逻辑中频繁使用的基础容器。

这些内建类型并不是简单的数据结构集合。PVM 还应当为它们实现各自的运行时行为，例如字符串连接、列表下标访问、字典查找、对象比较和真值判定等。因此，在 PVM 中实现内建类型，既要考虑对象布局，也要考虑相关运算和方法语义如何被统一组织和调度。

### 2.4.3 类的封装

封装的核心含义，是将对象状态与围绕该状态运行的方法组织在一起。Python 同样支持这一基本思想：用户可以在类中定义属性和方法，再由实例对象持有具体状态，通过方法对外提供行为。

不过，Python 的封装方式与 C++、Java 等语言并不完全相同。首先，Python 并不提供严格的访问控制修饰符，如 `public`、`private` 和 `private`。其次，Python 类自身在运行时具备高度的动态性；例如，在 Python 程序运行过程中，可以向一个类中动态添加新的方法，或者将类中的方法替换成另一个函数。最后，由于 Python 允许程序运行过程中向对象实例单独添加属性，因此对于对象方法的绑定由做出了进一步限制。例如，对于向对象实例单独添加的函数，将不会被 PVM 视为对象方法，因而不会与该对象实例发生绑定。

这意味着 PVM 在实现对象系统时，不能简单照搬静态语言固定内存布局的对象模型，而是需要采用字典化的方式存储对象实例属性和类属性，从而提供动态性。相应地，对象属性查找以及方法绑定机制需要通过查找字典的方式，得出正确结果。

### 2.4.4 类的继承

继承允许一个类在已有类的基础上扩展行为和结构，是 Python 面向对象机制中的重要组成部分。通过继承，子类可以复用父类的属性和方法，也可以重写父类方法形成新的行为。

除了单继承之外，Python 还支持多继承。多继承带来的核心问题是：当多个父类中存在同名属性或方法时，PVM 应当按照什么顺序进行查找。为此，Python 引入了方法解析顺序（Method Resolution Order，MRO）的概念，并采用 C3 线性化算法生成类层次结构中的查找顺序。

从 PVM 实现角度看，继承并不仅仅意味着“复制父类的属性与方法”，更关键的是建立一个可按确定规则生成的查找顺序。后续对象属性访问、方法绑定和内建运算分派，都可能依赖这个顺序。因此，必须将类型继承关系的表达和 MRO 查找顺序的生成机制的实现，纳入 PVM 对象系统设计之中。

### 2.4.5 类的多态

多态指相同的调用接口在不同对象上表现出不同的行为。对于 Python 来说，多态并不只来源于传统的继承和方法重写，也和动态类型语言常见的鸭子类型特征密切相关。只要一个对象提供了调用方所需的方法，它就可以被调用方使用。

这对 PVM 的实现提出了约束：虚拟机系统不能照抄 C++ 等静态语言的方法调用机制，在运行阶段之前就尝试将 Python 程序中的方法调用操作与特定的方法函数相绑定；Python 中的方法调用操作，必须在运行阶段依赖统一的属性查找和方法绑定机制进行。换句话说，Python 对象系统的实现通常需要把“对象表示”和“对象行为调用”区分开，使得多态分派通过独立的查找机制完成。

## 2.5 Python 中的异常机制

异常机制本质上是一种特殊的控制流机制。与普通的顺序执行或条件跳转不同，异常会在运行时打断当前指令流，并尝试将控制权转移到最近的异常处理器处。因此，对 PVM 而言，异常机制并不是附属功能，而是作为解释器主执行逻辑的组成部分存在。

Python 的异常处理以异常对象（如 `RuntimeError`、`TypeError` 等）为中心。当程序执行过程中虚拟机内部抛出异常，或用户显式执行 `raise` 语句时，PVM 会创建并传播一个异常对象，并在虚拟机内部记录当前的异常状态（如具体的异常对象、发生异常时解释器的程序计数器等）。

之后，PVM 需要查找并将控制流转向 Python 程序中与异常发生点相匹配的异常处理器。与 C++ 类似，Python 语言中这个过程是通过栈展开（stack unwinding）完成的，只不过是由 PVM 完成的。

栈展开的过程可以用伪代码进行描述：
```python
while 虚拟机调用堆栈非空:
  if 当前栈帧对应的函数中存在可匹配的 `except` 或 `finally` 处理器:
    将解释器控制流转入该处理器执行
    break
  将当前栈帧从虚拟机调用堆栈中弹出
```

自 CPython 3.11 起，异常处理引入了函数级异常表（exception table）。也就是说，解释器不再需要早期那种显式维护块栈的方式来描述异常处理范围，而是在异常发生后，由 PVM 通过当前栈帧对应函数的异常表，查询是否存在可匹配的处理器。这种方式使得正常执行路径几乎不需要为实现异常处理而付出额外开销，同时也让异常控制流与普通控制流之间的分工更清晰。

对一个兼容现代 CPython 字节码模型的 PVM 而言，异常机制至少要解决四个问题：如何表示异常状态，在虚拟机内部抛出异常后如何持续传播异常，如何根据异常表查找处理器，以及如何实现栈展开。在本节介绍的理论基础之上，在本文所实现的 S.A.A.U.S.O VM 中，异常系统正是围绕这几个问题展开设计的。

## 2.6 Python 中的模块机制

模块（module）机制是实际 Python 项目中组织代码和复用功能的重要方式。对用户而言，`import` 语句看起来很简单；但对 PVM 来说，模块导入涉及名称解析、模块查找、代码加载、模块体执行、导入缓存以及父子模块绑定等多个环节，是一个典型的系统级功能。 

从概念上看，模块既是一段代码的载体，也是一层命名空间边界。从 PVM 设计与实现的角度来看，解释器执行 `IMPORT_NAME`、`IMPORT_FROM` 等相关字节码时，运行时通常需要先判断目标模块是否已存在于缓存中；若不存在，则按照搜索路径（如 `sys.path`）查找模块文件或包入口，创建模块对象，并执行模块体初始化其命名空间。执行完成后，该模块会被登记到 `sys.modules`，供后续重复导入时复用。

对于包（package）来说，模块机制还需要额外处理父子层级关系、相对导入级别（level）以及 `from ... import ...` 的绑定语义。这也是为什么模块系统在实现上往往不能只写成一个简单的“打开 Python 代码文件并执行”过程，而需要拆分出查找、解析、加载和缓存等多个职责明确的阶段。

对本文而言，模块系统之所以值得在理论基础中单独讨论，是因为 S.A.A.U.S.O VM 并不仅仅支持运行一段孤立的脚本程序，还可以支持更接近真实 Python 程序结构的代码组织方式。只有具备最小可用的模块机制，虚拟机系统的完整度和可实际使用程度才能到达一个更高的台阶。

## 2.7 本章小结

本章首先探讨了 PVM 的执行原理，指出从执行模型上来看 PVM 后端在本质上是一个虚拟的栈机；然后逐一分析为了实现 Python 语言的核心功能，PVM 后端应当具备的能力或应解决的关键问题。

从整体上看，PVM 属于"前端与后端分离"执行结构：前端负责翻译 Python 源代码并生成字节码形式的程序表示；后端负责字节码的装载、调度、执行以及运行时语义支撑。对 PVM 后端而言，最关键的基础问题主要包括：如何组织字节码执行模型、如何管理函数调用与栈帧、如何表示 Python 对象及其类型关系、如何处理异常控制流，以及如何实现最小可用的模块加载机制。

本章的分析与讨论，为 S.A.A.U.S.O VM 的整体架构，以及关键模块的设计与实现提供了理论基础。

# 第3章 S.A.A.U.S.O VM 系统总体设计

第 2 章已经从理论层面说明了一个轻量级 Python 虚拟机后端所需要处理的几个关键问题，包括字节码执行、函数调用、对象模型、异常机制和模块导入等。本章在此基础上，进一步给出 S.A.A.U.S.O VM 的总体设计方案，重点回答“系统整体如何划分”“各模块如何协作”以及“为什么采用这样的结构”。

从工程目标上看，S.A.A.U.S.O VM 并不追求完整复刻 CPython 的所有能力，而是围绕“轻量、可嵌入、结构清晰、便于教学与二次开发”这一目标展开设计。因此，在总体架构上，该 VM 在设计时尽量遵循职责单一、边界明确的模块划分原则，使解释执行、对象表示、运行时语义和嵌入接口之间能够形成较清晰的协作关系。

## 3.1 设计目标与总体思路

S.A.A.U.S.O VM 的总体设计主要围绕以下几个目标展开。

首先，系统需要兼容 CPython 3.12 字节码模型中的核心执行路径，使常见的 Python 核心功能能够在不依赖完整 CPython 运行时的情况下被解释执行。这里的“兼容”并不意味着逐项复刻完整工业实现，而是以教学和嵌入场景为目标，优先支持最有代表性的语言语义和运行时机制。

其次，系统需要保持良好的可嵌入性。也就是说，S.A.A.U.S.O VM 不仅要能够作为独立解释执行系统运行脚本，还应当能够通过清晰的对外接口嵌入到宿主 C++ 程序中，作为脚本引擎使用。基于这一考虑，系统设计从一开始就保留了专门的 Embedder API 层，并将其与 VM 内核实现进行隔离。

再次，系统需要具备较好的教学可读性和工程可维护性。对于一个轻量级 PVM 后端而言，真正困难的部分往往不在于单个模块能否运行，而在于多个子系统之间能否形成稳定、清晰、可解释的协作方式。因此，本文没有采用“所有能力集中堆叠到解释器内部”的做法，而是尝试把对象系统、执行系统、运行时语义、模块系统和内存管理做成相对独立又彼此配合的层次结构。

基于上述目标，S.A.A.U.S.O VM 的总体设计采用了“前端与后端分离、内核与嵌入接口解耦、运行时状态集中管理、对象表示与行为分派分离”的思路。前端负责将源码编译为字节码或将 `.pyc` 文件解析为内部代码对象；后端负责解释执行、异常处理、对象管理和模块装载；对外则通过嵌入接口向宿主程序暴露脚本执行能力。

## 3.2 系统总体分层架构

从整体结构上看，S.A.A.U.S.O VM 可以分为前端、后端和嵌入接口三个大的部分。其中，前端负责代码装载与代码对象构建，后端负责运行时核心能力，嵌入接口则负责把这些能力以相对稳定、易用的形式暴露给宿主应用。

如果进一步按职责细分，系统内部可以概括为如下几层：

- 工具基础层（Utils）：提供与虚拟机业务无关的通用工具与基础设施；
- 句柄与安全引用层（Handles）：提供 `Handle`、`HandleScope` 等机制，用于在垃圾回收场景下安全持有对象引用；
- 对象表示层（Objects）：定义 Python 对象的统一表示方式，以及类型元信息、对象布局和核心对象模型；
- 执行门面层（Execution）：收口脚本执行、函数调用和当前运行状态访问等统一入口；
- 运行时语义层（Runtime）与内建函数层（Builtins）：承载跨类型语义逻辑与内建能力；
- 字节码解释执行层（Interpreter）：负责实际的字节码分派、栈帧推进、调用调度与异常展开；
- 模块系统（Modules）：负责 `import` 相关的名称解析、路径查找、模块加载与缓存管理；
- 代码装载前端（Code）：负责源码编译链路或 `.pyc` 解析链路；
- 嵌入接口层（Embedder API）：为宿主 C++ 程序提供使用 VM 的公共入口。

这种分层方式并不是简单按目录切分，而是服务于两个核心目标。其一，是降低模块耦合，避免解释器承担过多高层语义职责；其二，是为系统后续继续扩展语言特性、优化执行路径或加强嵌入接口打下更清晰的结构基础。

从依赖方向上看，系统整体遵循自底向上的单向依赖原则。底层对象表示和句柄安全机制为高层运行时提供基础；运行时和内建函数为解释器提供语义支持；执行门面将解释器、运行时和模块系统对外收口；嵌入接口则只与门面层和必要的桥接逻辑交互。这样的设计既能保持后端内核的相对稳定，也有利于控制复杂度。

## 3.3 核心运行时容器设计

在 S.A.A.U.S.O VM 中，`Isolate` 是最核心的运行时容器。它既可以理解为一个独立的虚拟机实例，也可以理解为一个完整的 Python 运行时上下文。系统中的堆、工厂、解释器、模块管理器、内建函数环境、异常状态以及各类类型元信息，都会被集中挂载在某个 `Isolate` 之下。

采用 `Isolate` 作为系统装配中心，有几个明显好处。第一，它把原本分散的运行时状态统一收敛到了单一容器中，减少了全局状态带来的管理难度。第二，它天然适合作为嵌入接口中的最外层抽象：宿主程序只需要创建、进入和销毁一个 `Isolate`，即可拥有一个相对独立的“Python 世界”。第三，这种设计也有利于后续扩展，例如多个隔离运行环境、不同上下文之间的相对隔离等。

围绕 `Isolate`，系统还建立了若干重要的辅助组件：

- `Heap`：负责对象分配、空间组织和垃圾回收；
- `Factory`：负责统一对象创建入口，屏蔽底层分配细节；
- `Interpreter`：负责字节码调度与函数调用；
- `ModuleManager`：负责模块缓存、搜索路径与导入门面；
- `ExceptionState`：负责保存当前待传播异常；
- `StringTable` 与各类 `Klass`/`TypeObject`：负责运行时常量和类型信息。

在这种结构下，系统中的许多关键操作都不是直接跨模块互相调用，而是围绕 `Isolate` 上下文展开。例如，解释器执行时需要访问 builtins 和模块状态，运行时抛错时需要写入异常状态，嵌入接口创建对象时需要使用当前 `Isolate` 的工厂和堆。这使得系统行为在工程上更容易追踪，也更便于统一管理生命周期。

## 3.4 核心模块划分与职责

虽然 S.A.A.U.S.O VM 在外部看起来是一个整体，但从实现角度看，它实际上由多个职责差异明显的子系统构成。将这些职责拆开，不仅有利于降低实现复杂度，也有利于在论文中清晰解释每个模块各自承担的任务。

### 3.4.1 对象系统

对象系统是整个 VM 的基础。S.A.A.U.S.O VM 采用“对象实例数据与类型行为分离”的设计思路：对象实例主要负责保存数据，而类型元信息和行为分派则由专门的类型描述结构承担。这样做的好处在于，解释器、运行时和内建函数不需要把行为逻辑硬编码在某一种对象上，而是可以通过统一的类型入口进行分派。

这一设计也使系统更容易支持 Python 的动态对象模型。因为在 Python 中，函数、类、模块、字符串、列表等都属于对象，它们虽然具体表现不同，但在 VM 内部需要拥有统一的表示和统一的调度入口。对象系统正是承担这一职责的核心模块。

### 3.4.2 解释执行系统

解释执行系统负责字节码分派、函数调用、栈帧推进和返回值处理，是虚拟机最直接体现“执行能力”的部分。S.A.A.U.S.O VM 在这一层中重点解决三个问题：如何高效完成字节码取指与调度，如何围绕函数调用构造和销毁栈帧，以及如何让解释器与异常系统、对象系统和模块系统协同工作。

需要强调的是，解释器本身并不直接承担所有高级语义。例如，很多跨类型的运行时规则并不适合直接写进解释器分派循环中，否则会导致解释器快速膨胀。因此，解释器更多负责“控制执行流程”，而高层语义则尽量交由运行时层和对象系统完成。

### 3.4.3 运行时与内建能力

运行时层主要负责承载那些跨对象、跨类型、跨场景复用的语义逻辑。例如对象反射、可迭代对象展开、字符串转换、执行辅助逻辑等，都更适合放在运行时层统一处理，而不是散落到各个对象实体类中。

与之配套的内建函数层，则负责向 Python 程序暴露最基本的语言能力。诸如 `print`、`len`、类型构造函数以及部分最小异常类型对象等，都会在系统初始化阶段被统一装配到运行时环境中。这一部分既是“语言能用起来”的基础，也是宿主和脚本共享统一行为的重要桥梁。

### 3.4.4 模块系统

模块系统负责处理 Python 中的导入语义。相比简单执行单文件脚本，支持模块机制意味着 VM 必须处理模块缓存、搜索路径、包结构、相对导入和父子模块绑定等问题。因此，S.A.A.U.S.O VM 没有把 `import` 简化为某个解释器分支中的零散逻辑，而是专门拆出了模块管理、名称解析、文件查找和模块加载等子职责。

从总体设计上看，模块系统并不是解释器的附属功能，而是一个相对独立但与解释器紧密协作的运行时子系统。解释器负责识别导入相关字节码并发起导入动作，模块系统则负责真正完成导入过程。

### 3.4.5 堆与内存管理

内存管理是 VM 能否长期稳定运行的关键。S.A.A.U.S.O VM 当前以新生代回收主路径为核心，建立了对象分配、根扫描和基本复制回收的最小闭环。在总体设计阶段，内存管理模块的重点不只是“回收对象”，更在于为对象系统、句柄系统和解释执行系统提供统一、可协作的内存语义。

与传统 C/C++ 程序中由程序员手工控制对象生命周期不同，VM 运行时中的对象大多由堆统一管理，因此对象分配方式、引用持有方式以及 GC 可见性都必须提前纳入总体设计。这也是为什么 S.A.A.U.S.O VM 在对象系统之外，还专门设计了句柄与作用域机制。

### 3.4.6 嵌入接口

S.A.A.U.S.O VM 的一个重要目标，是让它不仅能独立运行脚本，还能作为脚本引擎嵌入到宿主程序中。因此，系统在总体设计阶段就引入了独立的 Embedder API 体系。该体系对外提供 `Isolate`、`Context`、`Script`、`Value`、`TryCatch` 等概念，对内则通过桥接层与 VM 核心实现相连接。

采用这种“外部接口层与内部实现层分离”的设计，有助于在不暴露内部对象布局和实现细节的前提下，为宿主应用提供稳定、易理解的接入方式。这一点对于工程应用和后续 SDK 化演进都非常重要。

## 3.5 关键数据流与执行流程设计

总体设计不仅要说明模块划分，还需要说明模块之间如何协作。对于 S.A.A.U.S.O VM 而言，最关键的数据流主要包括脚本执行流、函数调用流、异常传播流和模块导入流。

### 3.5.1 脚本执行流

脚本执行流可以概括为“代码装载 -> 代码对象生成 -> 进入执行门面 -> 构建运行上下文 -> 解释器执行”。如果输入是源码，则系统需要先通过编译链路将其转化为字节码表示，再构造内部代码对象；如果输入是 `.pyc`，则系统可以直接进入解析和装载过程。之后，执行门面将代码对象交给解释器，由解释器结合当前上下文完成实际运行。

这一流程体现了系统设计中的一个基本原则，即前端与后端尽量解耦。前端负责把“源代码或字节码文件”转换成内部可执行表示，后端则专注于如何执行这些表示。这样既方便控制系统复杂度，也为后续扩展不同前端入口预留了空间。

### 3.5.2 函数调用流

函数调用流可以概括为“识别可调用对象 -> 进行参数绑定 -> 构建新栈帧 -> 执行被调函数 -> 返回并恢复调用方状态”。从实现角度看，函数调用是解释器内部最复杂的主路径之一，因为它不仅涉及执行权切换，还需要处理默认参数、关键字参数、扩展参数、闭包变量以及方法调用时的隐式接收者等问题。

在总体设计上，S.A.A.U.S.O VM 将“参数绑定”和“字节码执行”这两件事分离开来：前者在栈帧构建阶段尽量完成，后者再交由解释器主循环处理。这样可以使解释器分派逻辑保持相对干净，也使函数调用路径更容易维护。

### 3.5.3 异常传播流

异常传播流是整个系统设计中的另一条关键路径。其基本过程是：运行时某处发生异常后，异常对象被挂入当前 VM 的异常状态；解释器在后续调度中检测到待传播异常，再根据当前代码对象携带的异常表查找处理器；若本帧无法处理，则继续向上层调用帧展开，直到找到可用处理器或最终传播到宿主边界。

这一流程说明，异常系统并不是“附加在解释器外面的一层壳”，而是与执行系统深度耦合的核心机制。也正因如此，S.A.A.U.S.O VM 把异常状态设计为运行时共享状态，而不是局部变量式的临时结构，以保证 builtins、runtime 和 interpreter 都能参与异常的统一传播过程。

### 3.5.4 模块导入流

模块导入流可以概括为“名称解析 -> 缓存检查 -> 文件查找 -> 模块对象创建 -> 模块体执行 -> 写入模块缓存”。如果是相对导入，还需要先根据当前模块上下文解析完整模块名；如果目标是一个包，则还要处理 `__init__` 入口和包路径信息。

在总体设计阶段，这条流程说明了一个重要问题：模块导入并不是单条字节码即可完成的简单动作，而是一组运行时子流程的组合。正因如此，模块系统在 S.A.A.U.S.O VM 中被设计成相对独立的子系统，而不是解释器中的临时补丁逻辑。

## 3.6 设计原则与边界控制

为了保证系统在规模逐步扩张后仍然保持可维护性，S.A.A.U.S.O VM 在总体设计阶段明确了一些工程原则。

第一，遵循单向依赖原则。底层工具层不依赖 VM 上层语义；对象系统不直接依赖高层业务逻辑；运行时和内建层尽量不直接耦合解释器内部细节。通过这种分层约束，可以有效避免模块之间相互牵连、职责漂移的问题。

第二，强调状态集中管理。系统中最核心的运行时状态尽量收敛在 `Isolate` 中，由它统一持有堆、解释器、模块状态和异常状态等关键资源。这样既便于管理生命周期，也有利于为宿主程序提供更清晰的 VM 实例语义。

第三，强调对象引用安全。由于 VM 内部对象大多由堆统一管理，并可能受到垃圾回收影响，因此系统在总体设计上不鼓励直接在高层接口中传播裸对象指针，而是依赖句柄和作用域机制来保证引用安全。

第四，强调错误显式传播。对可能失败的路径，应当通过统一的返回值和异常状态机制向上传递错误，而不是让错误在不同层级被随意吞掉。这样的设计既有助于解释器内部调试，也有助于嵌入接口侧对错误进行感知和处理。

第五，强调“可用优先、逐步演进”。考虑到本文课题定位为轻量级 PVM 后端，系统总体设计优先保证关键功能形成闭环，而不是在早期就追求覆盖所有 Python 语义或完整工业级优化。这一取舍既符合教学与实验目标，也更适合本科毕业设计的实际节奏。

综上所述，S.A.A.U.S.O VM 的总体设计并不是简单把“解释器、对象、GC、模块”这些模块拼在一起，而是围绕一个统一的运行时容器、明确的分层结构和若干关键执行流组织起来的。也正因为总体设计阶段先明确了这些边界，后续第 4 章中的关键模块实现才有可能在复杂度可控的前提下逐步展开。

# 第4章 S.A.A.U.S.O VM 关键模块实现

第 3 章已经从总体上给出了 S.A.A.U.S.O VM 的架构划分、核心运行时容器以及关键数据流。本章将在此基础上，进一步说明系统中若干关键模块的具体实现思路。考虑到本文的目标是展示一个轻量级 PVM 后端如何形成完整闭环，而不是逐文件介绍全部源码，因此本章只选取最能体现系统设计特点和工程难点的几个部分展开：对象系统、字节码解释器、异常处理机制、模块系统、嵌入接口以及堆与垃圾回收。

需要说明的是，本章仍然不会把每一个实现细节都铺开到源码级别，而是重点说明各模块“为什么这样设计”“核心数据结构是什么”“关键执行路径如何组织”，以及这些实现如何共同支撑前文提出的总体目标。

## 4.1 对象系统实现

对象系统是 S.A.A.U.S.O VM 的实现基础。无论是整数、字符串、列表、函数、模块，还是用户定义的类与实例，从运行时视角看，最终都需要被统一表示为可由 VM 管理和调度的对象。如果缺少一个统一的对象系统，后续的解释执行、属性访问、方法调用、异常传播和模块装载都难以形成稳定的实现基础。

### 4.1.1 统一对象表示

S.A.A.U.S.O VM 采用“对象实例数据与类型行为分离”的实现思路。对象实例本身主要保存运行时状态和字段数据，而与类型相关的名称、继承关系、方法分派入口以及对象遍历逻辑，则由独立的类型描述结构承担。这样做的直接好处是：对象表示和对象行为不会被混杂在一起，解释器与运行时也不必针对每一种对象类型编写大量特判分支。

在这一设计下，VM 内部的基础对象可以理解为由两部分组成：其一是对象头和对象数据区，其二是该对象所关联的类型元信息。对象头中保存对象生命周期与运行时管理所需的基础信息，类型元信息则负责描述该对象“是什么”和“能做什么”。这种分离式设计与许多现代工业级虚拟机的做法是一致的，也更适合后续扩展新的内建类型和用户自定义类型。

### 4.1.2 `PyObject` 与 `Klass` 的分工

在具体实现上，S.A.A.U.S.O VM 中所有堆对象都统一建模为 `PyObject` 体系；而对象对应的类型元信息，则由 `Klass` 体系统一描述。前者负责对象层面的布局和字段组织，后者负责对象层面的行为分派与类型关系管理。

这种实现方式带来的一个重要收益，是运行时可以将“对象的状态”和“对象的行为”区分开来。例如，一个字符串对象和一个列表对象在内存布局上显然不同，但它们都可以通过各自关联的 `Klass` 对象，向解释器和运行时暴露统一的行为入口。这样一来，解释器在执行属性访问、运算或调用时，不需要直接依赖具体对象类的细节，而是可以先通过类型元信息找到统一的处理路径。

同时，`Klass` 还负责组织类型继承关系、MRO、类型对象以及部分 GC 相关能力。这使得对象系统不仅能服务于语言语义，也能服务于内存管理和运行时调度。

### 4.1.3 基于 slot/vtable 的行为分派

仅仅有“对象 + 类型”还不足以支撑 Python 语义。要让对象真正参与运行，VM 还必须解决“如何统一调度行为”的问题。为此，S.A.A.U.S.O VM 在 `Klass` 基础上进一步引入了 slot/vtable 风格的行为分派机制。

具体来说，和对象行为相关的关键操作，例如属性读取、属性写入、下标访问、函数调用、类型构造、迭代、对象遍历以及实例大小计算等，都不会直接散落在解释器各处，而是通过统一的 slot 入口暴露出来。这样做至少有三个好处。

第一，它使运行时调用路径更加统一。解释器遇到某类字节码时，不需要为每种对象类型写一套独立分支，而是可以通过当前对象的类型信息找到对应 slot 执行。第二，它有利于新类型扩展。新增一个内建类型或用户类型时，只要实现相应 slot 并正确注册到类型描述中，就可以自然接入现有解释器和运行时。第三，它还能让 GC 相关能力与语言语义共享同一套类型入口，例如对象遍历函数和实例大小计算也可以作为 slot 的一部分被统一组织。

从工程角度看，这种设计使对象系统真正成为一个“运行时基础设施”，而不是若干分散对象类的集合。

### 4.1.4 类型对象、继承与 MRO

Python 对象系统的一个重要特征，是“类本身也是对象”。因此，S.A.A.U.S.O VM 在实现对象系统时，并没有把用户自定义类与内建类型完全割裂开来，而是通过类型对象与 `Klass` 的双向关联，把“类作为语言对象”和“类作为 VM 类型描述”统一到同一套框架中。

在继承关系处理上，系统遵循 Python 的多继承语义，并使用 MRO 作为属性查找与行为分派的重要依据。对于一个用户定义类而言，其最终行为并不只由自己定义的方法决定，还取决于其祖先类型、MRO 查找顺序以及类型对象上的相应属性和方法。因此，MRO 并不是可有可无的附加信息，而是支持类继承、方法查找和多态行为的关键组成部分。

在实现层面，S.A.A.U.S.O VM 把类型关系和 MRO 信息纳入 `Klass` 体系进行统一维护，这样解释器和运行时在进行属性查找、方法调用或类型判断时，都可以依赖同一套类型数据结构。

### 4.1.5 句柄与对象引用安全

由于 S.A.A.U.S.O VM 中的对象大多由堆统一管理，并可能在垃圾回收过程中发生移动，因此在高层模块之间直接传播裸对象指针会带来较大风险。为此，系统引入了 `Tagged`、`Handle` 和 `HandleScope` 等机制，用于在运行时安全持有对象引用。

这一设计体现了对象系统与 GC 之间的紧密关系。对象系统并不只是“定义对象长什么样”，它还必须回答“对象在回收过程中如何被安全引用”这一问题。对于一个教学导向的轻量级 PVM 后端来说，把这一层安全性机制内建到对象系统基础设施中，比把它留给上层模块零散处理更合理，也更能体现现代 VM 的工程实现思路。

## 4.2 字节码解释器实现

如果说对象系统为整个 VM 提供了统一的数据基础，那么字节码解释器就是系统真正“跑起来”的核心。S.A.A.U.S.O VM 的解释器模块负责接收代码对象和运行上下文，逐条调度字节码，并在执行过程中维护栈帧、调用关系、返回值和异常传播状态。

### 4.2.1 解释器总体结构

S.A.A.U.S.O VM 并没有把脚本执行入口直接暴露为“随便调用某个解释器方法”，而是先通过执行门面层统一收口，再由门面转入解释器内部。这种做法有两个考虑。其一，是避免高层模块直接耦合解释器内部细节；其二，是为脚本执行、函数调用、运行时辅助执行等不同调用场景保留统一入口。

在内部，解释器主要围绕三个核心要素展开：

- 当前正在执行的代码对象；
- 当前栈帧及其局部执行状态；
- 字节码分派循环与相关临时执行状态。

对解释器来说，真正困难的部分并不是“逐条执行字节码”这件事本身，而是在执行过程中同时正确处理函数调用、参数绑定、异常跳转、返回值回传以及与运行时层的协作。

### 4.2.2 `FrameObject` 与执行现场管理

为了支撑函数调用、递归执行和局部变量隔离，S.A.A.U.S.O VM 在解释器中引入了专门的栈帧对象。每个栈帧都对应一个当前正在执行的函数或顶层代码单元，负责保存该执行上下文中的程序计数器、操作数栈、局部变量、全局环境以及与代码对象相关的辅助信息。

采用栈帧对象而不是若干分散的局部变量结构，有几个直接收益。首先，它便于统一管理不同调用层级上的执行状态，使递归调用与普通调用能够共享同一套执行模型。其次，栈帧对象本身也可以被纳入 GC 根集合，使其中保存的局部变量和临时对象在回收过程中保持可达。再次，异常传播和函数返回都需要依赖栈帧边界来恢复控制流，因此用显式栈帧对象来表示执行现场更容易与其他模块协作。

### 4.2.3 字节码分派机制

在字节码执行阶段，S.A.A.U.S.O VM 采用基于调度表的分派方式组织解释器主循环。每次执行时，解释器会根据当前字节码的操作码定位对应处理入口，再由该入口负责完成具体语义。这一思路本质上仍然遵循传统的“取指—分派—执行”模型，但相比使用大量 `switch-case` 嵌套的简单做法，调度表方式更适合在字节码种类逐步增加后维持实现清晰度。

对一个轻量级 PVM 而言，解释器分派机制既关系到性能，也关系到可维护性。若所有语义都堆叠在一个巨大的分支结构中，后续扩展或排查问题都将变得困难。因此，S.A.A.U.S.O VM 在这部分实现中尽量将不同字节码语义拆到更清晰的 handler 路径中，并辅以运行时层和对象系统提供的统一辅助入口。

### 4.2.4 参数绑定与调用协议

函数调用是解释器最复杂的核心路径之一。S.A.A.U.S.O VM 在实现中并未把参数绑定逻辑直接揉进每一个调用字节码处理分支，而是引入专门的栈帧构建与参数归一化过程，使“调用准备”和“字节码执行”这两件事相对分离。

这样做的原因在于，Python 函数调用语义本身就相当复杂。位置参数、关键字参数、默认参数、`*args`、`**kwargs`、绑定方法中的隐式接收者，以及闭包变量参与的函数对象构造，都会影响最终如何建立一个新的可执行栈帧。如果这些细节全部压到解释器主循环中，不仅会使解释器逻辑快速膨胀，也会降低可读性。

因此，S.A.A.U.S.O VM 将调用协议的复杂性集中在栈帧构建阶段尽量消化，而让解释器主循环更多负责“在栈帧已经准备好之后，如何继续执行”。这一划分也与前文第 3 章中的“职责分离”总体设计保持一致。

### 4.2.5 解释器与运行时层的协作

需要特别强调的是，S.A.A.U.S.O VM 的解释器并不试图独自完成全部语义。很多高层运行时规则，例如跨类型对象转换、内建类型的复合语义、反射能力、导入辅助逻辑等，都更适合由专门的运行时层承担。解释器的主要职责是维持控制流推进、调用调度和执行状态管理，而不是成为所有语义规则的唯一落点。

这种做法虽然会使“解释器 + 运行时”的边界划分稍微复杂一些，但从长期看更有利于控制解释器规模，也更有利于在第 4 章这种实现说明中把“执行算法”和“语义实现”分开叙述。

## 4.3 异常处理机制实现

异常机制是 S.A.A.U.S.O VM 中最能体现“解释器并非简单顺序执行器”的一部分。与普通分支或循环不同，异常会主动打断当前执行路径，并要求系统重新寻找控制流目标。这意味着异常处理必须跨越解释器、运行时、栈帧和类型系统共同完成。

### 4.3.1 异常状态的统一管理

S.A.A.U.S.O VM 采用将异常状态集中挂载在 `Isolate` 上的实现方式，而不是把异常状态仅仅作为解释器的局部变量。这种设计有很强的工程合理性。因为异常并不只会在解释器的某条字节码处理中被创建，运行时层、内建函数层乃至嵌入接口桥接层，都可能触发异常对象的创建与传播。

如果异常状态只存在于解释器内部，那么其他模块在抛错时就必须依赖解释器的具体实现细节，这会破坏系统分层。将异常状态提升为运行时共享状态后，任何需要抛错的模块都可以通过统一路径写入 pending exception，而解释器再在合适的时机接管其传播过程。这样既保持了架构边界，也使错误传播逻辑更加一致。

### 4.3.2 异常对象与抛错门面

在具体实现中，S.A.A.U.S.O VM 并不是简单用一个字符串或错误码表示异常，而是按照 Python 语言模型将异常表示为对象。这样做有两个原因。第一，Python 语言本身就是以异常对象作为控制流载体，异常匹配与异常传播都依赖对象类型。第二，只有将异常表示为对象，解释器和宿主接口才能在后续更自然地支持异常捕获、异常转发和异常信息查询。

围绕这一点，系统实现了统一的抛错门面。无论错误源自运行时辅助逻辑、内建函数调用，还是解释器执行某条失败路径，最终都会尽量通过统一接口构造异常对象并写入异常状态。这种做法使异常处理从工程上具备更好的可维护性，也避免了不同模块各自定义错误格式而造成的混乱。

### 4.3.3 基于异常表的处理器查找

与较早期的简单异常实现不同，S.A.A.U.S.O VM 在异常传播主路径上采用了与现代 CPython 字节码模型相一致的异常表思路。也就是说，解释器在异常发生后，不是单纯依靠一套显式维护的块栈来查找处理器，而是结合当前代码对象中携带的异常表信息，判断当前指令地址是否落在某个处理范围内，并据此定位 `except` 或 `finally` 处理器。

这种实现方式的优势在于，正常执行路径上的额外负担较小，而异常处理逻辑则集中在异常真正发生时再进行。对本文这种教学导向但又希望贴近现代 CPython 字节码模型的轻量级 PVM 来说，这是一种较有代表性的设计选择。

### 4.3.4 跨栈帧异常展开

异常传播并不总能在当前函数内部结束。当当前栈帧中找不到可以处理异常的处理器时，解释器必须向调用栈更外层继续回溯。这意味着异常机制天然依赖栈帧结构：它必须知道当前代码对象对应的处理范围、当前帧的恢复方式、是否需要调整操作数栈深度，以及何时将控制权交回上一层调用者。

S.A.A.U.S.O VM 在这一部分实现中的关键点，是把“异常状态的记录”和“异常控制流的恢复”分开。前者由运行时共享状态承担，后者则由解释器在合适的执行阶段完成。这样的分工既让异常系统对其他模块保持开放，也让真正的异常展开逻辑能够继续留在解释器控制之下。

## 4.4 模块系统实现

模块系统是 S.A.A.U.S.O VM 从“能够执行单段脚本”走向“能够支撑较真实程序结构”的关键一环。相比独立运行一段顶层代码，支持模块导入意味着系统必须回答更多问题：模块名如何解析、文件如何查找、包结构如何识别、模块如何缓存、父子模块关系如何维护。

### 4.4.1 模块系统的职责拆分

在实现上，S.A.A.U.S.O VM 并没有把模块导入写成一个大而全的函数，而是根据导入流程中的不同职责，将模块系统拆分为若干相对独立的子部分，例如模块状态管理、导入编排、名称解析、模块定位和模块加载等。这种拆分方式与前文的总体设计思路一致，都是为了避免单一模块承担过多不属于自己的职责。

从工程角度看，`import` 语义本身就是一个组合动作：它既涉及语言层的导入语义，也涉及文件路径、缓存复用和模块对象初始化。因此，拆分模块系统并不是额外增加复杂度，而是在控制复杂度。

### 4.4.2 模块管理与缓存

S.A.A.U.S.O VM 中的模块系统以统一的模块管理器为入口，负责维护类似 `sys.modules` 与 `sys.path` 的核心状态。前者承担模块缓存职责，后者承担模块搜索路径职责。解释器在执行导入相关字节码时，会先通过模块管理器判断目标模块是否已经加载，若已经存在，则直接复用缓存结果；否则再进入后续查找与加载流程。

这种缓存机制对于 Python 语义非常重要。它不仅能避免重复执行同一个模块体，也有助于维持模块对象身份一致性。对模块导入而言，缓存并不是性能上的小优化，而是语义正确性的一部分。

### 4.4.3 名称解析与模块查找

模块导入并不总是简单地给出一个绝对模块名。Python 支持相对导入、包导入以及 `from ... import ...` 等多种形式，因此系统必须先把这些形式解析为明确的目标模块，再根据搜索路径去定位对应文件或包入口。

S.A.A.U.S.O VM 在实现中将这一过程划分为两个阶段：第一阶段先解析完整模块名，第二阶段再根据模块名在搜索路径中查找可能的文件位置。这样做的优点是，可以把“Python 语义层面的名称解释”与“文件系统层面的路径定位”分开处理，使模块系统更容易维护和扩展。

### 4.4.4 模块加载与执行

当模块位置确定后，系统需要创建对应的模块对象，补全其命名空间中的必要字段，并执行模块体代码完成初始化。这里的执行本质上仍然是“把模块代码对象交给解释器运行”，只不过其上下文不再是普通脚本的全局环境，而是当前模块自己的命名空间。

这一过程再次体现出对象系统、解释器和模块系统之间的协作关系：模块对象由对象系统表示，模块体执行由解释器完成，而导入时机和缓存策略则由模块系统调度。也正是因为这些职责被拆分清楚，模块机制才能较自然地嵌入 VM 整体结构中。

## 4.5 Embedder API 实现

除了作为独立解释执行系统之外，S.A.A.U.S.O VM 还希望具备较好的嵌入能力。因此，系统专门设计并实现了一套面向宿主程序的 Embedder API，用于把 VM 核心能力以更稳定、易用的方式暴露给外部 C++ 应用。

### 4.5.1 对外接口的总体结构

S.A.A.U.S.O VM 的嵌入接口采用“公共头文件 + API 桥接层 + VM 内核实现”三层结构。最外层是供宿主直接包含和调用的公共接口，中间层负责把公共接口对象转换为 VM 内核可理解的内部表示，最内层则是真正的解释器、运行时和对象系统。

这种结构的最大优点是实现细节不会直接泄漏给外部调用者。宿主程序可以通过 `Isolate`、`Context`、`Script`、`Value`、`Function`、`TryCatch` 等相对稳定的概念使用虚拟机，而不需要了解内部对象布局、GC 细节或解释器状态管理方式。

### 4.5.2 运行时生命周期管理

在 Embedder API 中，`Isolate` 是最外层的 VM 生命周期单元。宿主程序首先初始化库并创建 `Isolate`，再在其内部创建 `Context` 作为脚本运行环境，最后通过 `Script` 接口编译和执行脚本。这样的结构与前文总体设计中“以 `Isolate` 作为系统装配中心”的思路保持一致，也使对外接口具有较好的可理解性。

从宿主视角看，这套接口大幅降低了“接入一个虚拟机”所需的心智负担：只要理解一个 VM 实例、一个上下文环境和一段脚本对象这三个核心概念，便能基本完成嵌入过程。

### 4.5.3 宿主与脚本的双向互操作

S.A.A.U.S.O VM 的 Embedder API 并不只支持“宿主把一段脚本跑起来”，还支持更重要的双向互操作能力。一方面，宿主程序可以创建函数对象并将其注入脚本环境，使 Python 脚本调用 C++ 回调；另一方面，宿主程序也可以从脚本上下文中取回 Python 函数对象，并在 C++ 侧发起调用。

这种双向互操作能力，是 S.A.A.U.S.O VM 具备“脚本引擎”属性的重要体现。对论文而言，这一部分不仅展示了嵌入接口的工程价值，也展示了对象系统、函数调用机制和异常传播机制如何向外部接口自然延伸。

### 4.5.4 失败语义与异常观测

为了让嵌入接口在错误场景下保持可控，S.A.A.U.S.O VM 没有简单采用“失败即崩溃”的方式，而是通过显式的失败返回值与 `TryCatch` 观察器共同组织 API 边界上的错误处理。这使宿主程序能够区分“调用失败”“异常已被抛出”“返回值为空”等不同状态，并根据需要继续做错误处理或恢复。

从设计理念上看，这与 VM 内部强调“错误显式传播”的原则是一致的。也就是说，Embedder API 并不是独立发明了一套新的失败语义，而是把 VM 内核已有的错误传播思路，在对外接口上做了更适合宿主使用的封装。

## 4.6 堆与垃圾回收实现

对于一个基于堆管理对象生命周期的虚拟机而言，内存管理不是一个后加模块，而是整个系统能否稳定运行的基础能力。S.A.A.U.S.O VM 当前在这部分的实现目标，是建立一个以新生代回收为主的最小可用闭环，使解释器、对象系统、句柄系统和运行时都能够在统一的托管内存语义下协作。

### 4.6.1 堆空间划分

S.A.A.U.S.O VM 在总体上对堆进行了分空间组织，主要包括新生代空间、元数据空间以及预留的老生代相关结构。这样的设计并不是为了追求一开始就实现完整工业级分代 GC，而是为了给对象生命周期、元数据驻留和后续回收策略扩展预留清晰边界。

其中，新生代空间主要用于分配普通运行时对象，元数据空间则主要承载较长期驻留的类型元信息、字符串常量等对象。这样的划分有助于降低运行时对象和稳定元数据之间的相互干扰，也为后续更细粒度的 GC 策略演进提供基础。

### 4.6.2 根集合与对象可达性

GC 能否正确工作，关键不只在于“如何回收”，更在于“如何找到不能回收的对象”。在 S.A.A.U.S.O VM 中，根集合不仅来自显式全局状态，还来自多个核心运行时模块，例如当前 `Isolate` 持有的关键对象、各类类型元信息、模块缓存、异常状态、解释器当前栈帧以及活动中的句柄作用域等。

这意味着垃圾回收不是堆模块自己的孤立问题，而是整个 VM 共同参与的过程。如果根集合组织不正确，那么即使回收算法本身没有问题，也可能发生对象丢失、悬垂引用或错误回收。因此，S.A.A.U.S.O VM 在实现中把对象遍历、句柄作用域和运行时状态管理放在同等重要的位置考虑。

### 4.6.3 当前回收策略与工程取舍

在当前阶段，S.A.A.U.S.O VM 主要实装的是以新生代为核心的 Scavenge 回收主路径。其基本思想是：新分配对象首先进入新生代空间，回收时通过复制存活对象并更新相关引用来完成清理。这种策略实现相对直接，适合作为轻量级 VM 的第一阶段回收方案。

采用这一策略的原因主要有两点。第一，它足以支撑当前对象系统、解释器和模块系统形成可运行的最小闭环。第二，相较于一开始就实现完整老生代回收、写屏障和 remembered set 等更复杂机制，新生代主路径更适合作为可验证、可渐进扩展的实现起点。

### 4.6.4 工厂分配与对象初始化约束

在对象创建方面，S.A.A.U.S.O VM 没有把堆分配逻辑直接散落到各个业务模块中，而是通过统一工厂入口组织对象分配与初始化。这种设计有助于统一控制对象创建顺序、字段初始状态以及与 GC 的协作边界。

尤其是在对象尚未完成初始化时，如果发生新的堆分配或触发 GC，往往容易导致对象进入“半初始化但已暴露”的危险状态。因此，系统在实现中对对象分配与初始化阶段进行了额外约束，尽量把“分配”“字段填充”“注册可达性”几个步骤组织得更清晰。这也是轻量级 VM 在工程实现上经常容易被忽视、但实际非常关键的部分。

## 4.7 本章小结

本章围绕 S.A.A.U.S.O VM 的若干关键模块，对其实现思路进行了归纳和说明。可以看到，虽然对象系统、解释器、异常、模块系统、嵌入接口和垃圾回收在表面上属于不同方向的问题，但它们在实现上并不是彼此孤立的：对象系统为解释器和模块系统提供统一数据表示，解释器依赖栈帧和异常系统完成执行控制，模块系统依赖对象系统和解释器完成模块加载，嵌入接口则把这些内核能力进一步整理为对宿主可用的脚本引擎接口。

因此，S.A.A.U.S.O VM 的实现难点并不在于单独实现某一个模块，而在于让这些模块在统一的运行时容器和清晰的分层结构下形成稳定协作关系。正是这种跨模块的系统性实现，构成了本文工作的核心工程内容。下一章将在此基础上，通过测试与结果分析对系统的功能正确性、兼容性和当前表现进行验证。
