# 计划：补齐内建类型 `__init__` Python 可见性并对齐 CPython 3.12

## 目标与验收标准

* 补齐 S.A.A.U.S.O 中“应在 Python 世界可访问”的内建类型 `__init__` 暴露能力，重点覆盖 `list/dict/str/tuple` 等当前存在缺口的类型访问路径。

* 使 `list.__init__` 作为类型属性被读取后仍可调用，支撑自定义类中显式调用基类初始化。

* 重新启用并通过 `test/unittests/interpreter/interpreter-custom-class-unittest.cc` 第 583-620 行对应的两个测试用例。

* 新增并通过补充单测，覆盖更多内建类型 `__init__` 调用语义与错误语义，行为尽量与 CPython 3.12 对齐。

* 完成构建与相关单测验证，确保无回归。

## 现状判断（基于代码调研）

* 当前实例化主路径已统一为 `Runtime_NewObject: __new__ -> __init__`，且会校验 `__init__` 返回 `None`。

* `BaseException.__init__` 已通过方法表显式暴露；但 `list/dict` 等主要通过 C++ 槽位 `InitInstance` 实现，未系统注入到类字典可见层。

* 类型属性查找当前主要依赖类字典/MRO 属性查找，导致 `list.__init__` 这类“从 type 取属性再调用”的路径存在缺口。

## 实施步骤

1. 明确“哪些类型需要暴露 `__init__`，哪些应保持仅 `__new__` 语义”

   * 逐一核对 `list/dict/str/tuple/int/float/object/type/bool` 的 CPython 3.12 语义与当前实现注释。

   * 产出项目内一致规则：可暴露集合、仅 `__new__` 集合、对未定义 `__init__` 的回退策略。

2. 设计并落地 `__init__` 属性桥接方案（首选统一层，不做散点补丁）

   * 在现有类型属性访问链路中引入“槽位到 Python 可调用对象”的桥接入口，优先复用既有 `__init__` 桥接基础设施与错误传播风格。

   * 确保桥接对象读取后可重复调用，不破坏实例化路径中的 `InitInstance` 语义与 `None` 返回校验。

   * 保持 MRO/override 语义一致，不引入对虚函数重写判定的额外破坏。

3. 覆盖关键调用路径并处理边界行为

   * 覆盖 `list.__init__(obj, iterable)`、`dict.__init__(obj, mapping/iterable)` 等典型路径。

   * 明确并实现参数错误、接收者错误、返回值错误等异常路径，保证与现有异常体系/MaybeHandle 契约一致。

   * 验证对“仅 `__new__` 类型”（如 `int/float/tuple`）不产生错误暴露。

4. 先恢复并增强现有回归测试

   * 取消注释并修复第 583-620 行两个测试（基类 `list.__init__` 调用、自属性加载后可调用性）。

   * 对期望输出与异常断言使用现有测试 helper 规范，避免重复样板。

5. 新增补充单测（同目录下 interpreter 相关单测）

   * 新增“内建 `__init__` 可见性”测试：确认 `list.__init__`、`dict.__init__` 在 Python 层可取用并可调用。

   * 新增“行为一致性”测试：验证调用后对象状态更新、重复调用行为、错误参数时报错类型/关键文案。

   * 新增“边界一致性”测试：确认仅 `__new__` 类型不会被错误地当成可用 `__init__` 入口。

6. 编译与测试验证

   * 完整编译当前工程目标（沿用仓库既有构建方式与严格编译选项）。

   * 重点执行并确认：`interpreter-custom-class-unittest.cc#L583-620` 对应用例通过。

   * 执行新增用例与受影响解释器单测集合，确认无回归。

7. 收尾与结果整理

   * 总结改动点（桥接入口、类型范围、测试覆盖面）。

   * 给出“与 CPython 3.12 已对齐项 / 仍待后续增强项（如 override 更深语义）”的清单，便于后续迭代。

## 风险与控制

* 风险：在类型属性查找链路做桥接可能影响其他 dunder 属性解析。

  * 控制：限定仅 `__init__`（必要时含 `__new__`）桥接路径，增加回归测试覆盖非目标属性访问。

* 风险：`tuple/int/float` 等仅 `__new__` 语义被误改。

  * 控制：在类型名单与桥接条件中显式白名单/黑名单控制，并新增负向测试。

* 风险：异常信息文案与 CPython 不完全一致导致测试脆弱。

  * 控制：测试以“异常类型 + 关键子串”断言为主，减少对全文案硬绑定。
