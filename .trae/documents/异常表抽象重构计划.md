# 异常表（ExceptionTable）抽象与解耦：重构计划

## 结论：你的建议是否合理

合理且值得做。

* 当前把 `co_exceptiontable` 作为 `PyString` 直接暴露给解释器，并把 varint 解码 + handler 查找逻辑硬编码在 `interpreter-dispatcher.cc`，会让解释器主循环承担过多“字节码元数据格式细节”，削弱单一职责与可测试性。

* 抽象 `ExceptionTable` 类后，解释器只关心“给定 code object + raising IP，查询到 handler/stack-depth/push-lasti”，格式细节收敛在一个独立模块里，更符合仓库既有的“runtime helper / interpreter 执行层”分工。

## 重构目标

* 降低 `Interpreter`/dispatcher 文件复杂度，让解释器主循环更聚焦于“调度与栈帧执行”。

* 将异常表格式（CPython3.11+ zero-cost exception table varint 编码）解码与查找逻辑集中封装，形成可复用/可单测的组件。

* 保持当前行为与性能特征：不改变 `.pyc` 解析产物、不改变 `PyCodeObject` 内部字段表示（仍然是 `PyString` 存 bytes），不改变 try/except/finally 单测结果。

## 非目标（本轮不做）

* 不把 `exception_table_` 从 `PyString` 改为 `PyBytes`/自定义对象（这会牵涉对象系统与 marshal/GC 迭代规则调整，可在后续独立治理）。

* 不引入更完整 traceback/错误文案系统（仍按当前 MVP 异常机制推进）。

* 不做 exception table 的二分查找/缓存优化（先完成职责解耦；后续再按性能需要迭代）。

## 设计草案

### 1) 新增 ExceptionTable 类

位置建议：`src/interpreter/exception-table.{h,cc}`（属于解释器执行元数据的实现细节；不放到 `src/utils`，因为它依赖对象/handles）。

对外 API（建议）：

* `struct ExceptionHandlerInfo { int stack_depth; int handler_offset_in_bytes; bool push_lasti; };`

* `class ExceptionTable final {`

  * `static bool LookupHandler(Handle<PyCodeObject> code_object, int ip_in_bytes, ExceptionHandlerInfo& out);`

  * `};`

封装点：

* varint 解码（当前 `ParseExceptionTableVarint`）

* 顺序扫描查找（当前 `FindExceptionHandler`）

* `ip` 单位换算（code units ↔ bytes）在类内统一处理

### 2) Interpreter 侧改动

* 删除 `interpreter-dispatcher.cc` 中匿名命名空间里的：

  * `ParseExceptionTableVarint`

  * `FindExceptionHandler`

* 在 `pending_exception_unwind` 路径中改为调用 `ExceptionTable::LookupHandler(...)`，解释器只消费 `ExceptionHandlerInfo`。

### 3) 构建系统接入

* 将新增的 `exception-table.cc` 加入根 `BUILD.gn` 的 `saauso_core.sources`（或当前解释器 source\_set 所在位置），确保 `vm/ut` 均可链接。

## 具体实施步骤（执行阶段）

1. 新增 `src/interpreter/exception-table.h/.cc` 并实现 `LookupHandler`
2. 更新 `src/interpreter/interpreter-dispatcher.cc`：

   * 引入新头文件

   * 移除旧的 helper 函数

   * 用 `ExceptionTable::LookupHandler` 替换原查表逻辑
3. 运行 `ut`：

   * 至少跑异常相关用例过滤（`BasicInterpreterTest.TryExcept*`）

   * 全量 `ut`（若时间允许，建议完整回归）

## 验收标准

* 代码结构上：exception table 格式细节不再出现在解释器 dispatcher 文件中。

* 行为上：异常机制相关单测全部通过，且无额外行为变化。

* 依赖上：不引入 `src/utils` 对 VM 上层的依赖；模块边界清晰。
