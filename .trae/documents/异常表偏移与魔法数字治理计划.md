# 异常表偏移与魔法数字治理计划

## 背景与目标

当前解释器异常展开链路已接入 `co_exceptiontable`（CPython 3.11+），但在实现上存在两类高风险问题：

1. **单位/偏移不变量容易被破坏**：异常表中的 `start/size/target` 与解释器 `pc` 的计量单位若不一致，会出现“明明存在 handler 却查不到”的隐蔽 bug。
2. **`exception-table.cc`** **大量 magic number**：影响可读性与可维护性，也增加未来改动时误用常量的风险。

本计划的目标是：

* 把异常表解析与查表逻辑中的“单位”显式化，并提供清晰的不变量与防护。

* 清理 `exception-table.cc` 中的意义不明数字，改为 `constexpr` 常量与/或强类型枚举，并通过小型 helper 函数表达语义。

* 将“禁止意义不明 magic number”固化进 `AGENTS.md` 的代码规范，便于后续代码与 AI 生成代码统一遵守。

## 范围

* 代码：`src/interpreter/exception-table.{h,cc}`，以及必要的调用点（主要是解释器异常展开处的调用与参数单位约定）。

* 文档：`AGENTS.md` 的代码规范章节。

* 测试：优先补齐/增强解释器端到端异常用例或最小单元测试，以锁定单位不变量。

## 方案总览

### 1) 明确并固化异常表“单位/偏移”不变量（核心治理）

#### 1.1 明确目前各处的单位

* `ExceptionTable::LookupHandler(..., instruction_offset_in_bytes, ...)` 当前 API 名义上使用 **byte offset**。

* `co_exceptiontable`（CPython 3.11+）表项的 `start/size/target` 是 **code units**（与 CPython “wordcode” 对齐，通常 1 code unit = 2 bytes）。

* 解释器在异常展开处把 `pending_exception_pc` 设置为 `current_frame_->pc() - kBytecodeSizeInBytes`，看起来 `pc()` 是 **bytes**，且 `kBytecodeSizeInBytes` 常量也表达了“每条指令字节宽度”的不变量。

风险点在于：目前 `exception-table.cc` 内部直接 `start *= 2; size *= 2; target *= 2;`，把单位转换“埋”在实现细节里，未来只要：

* bytecode 宽度变更（不再是 2 bytes），或

* `pc()` 单位调整（改为 code units），或

* pyc parser/解释器表项含义理解偏差，

都会让查表静默失效。

#### 1.2 固化单位：引入显式类型别名与转换 helper

在 `exception-table.cc` 的匿名命名空间中引入（严格限制作用域）：

* `using ByteOffset = int;`

* `using CodeUnitOffset = int;`

* `constexpr int kCodeUnitSizeInBytes = 2;`（与当前 CPython3.12 wordcode 对齐）

* `constexpr ByteOffset CodeUnitsToBytes(CodeUnitOffset x)`：在转换处做溢出检查（用 `int64_t` 中间值），并集中表达“为什么乘以 2”。

并将局部变量改名为更自解释的语义：

* `start_units / size_units / target_units`（code units）

* `start_bytes / end_bytes / target_bytes`（bytes）

同时将 `LookupHandler` 的参数名从 `instruction_offset_in_bytes` 保留为 byte 语义，并在函数头部用简短中文注释写清楚：

* 输入：byte offset

* 表：code units

* 内部统一转换为 bytes 后比较

#### 1.3 增加防护性校验（Debug/Release 都有效）

在不改变现有对外行为的前提下补充：

* 解析出的 `start/size/target` 必须非负，否则返回 false（视为表损坏）。

* `end = start + size` 溢出检查（用 64-bit 中间值）。

* 若未来 `kBytecodeSizeInBytes` 不为 2，单测会立刻失败，避免“暗改导致查表错位”。

（如需更强约束）可在调用点补充断言：`pending_exception_pc` 必须与 `kBytecodeSizeInBytes` 对齐（例如必须是 2 的倍数），但该断言是否会误伤 future 的“可变长字节码”方案，需要在实现阶段评估。

### 2) 清理 `exception-table.cc` magic number（可读性治理）

对 `ParseExceptionTableVarint` 与 `LookupHandler` 中的 magic number 分两类治理：

#### 2.1 varint 编码常量抽取为 `constexpr`

在匿名命名空间提取：

* `constexpr uint8_t kVarintPayloadMask = 0x3F;`（63）

* `constexpr uint8_t kVarintContinueMask = 0x40;`（64）

* `constexpr int kVarintPayloadBits = 6;`

并把：

* `b & 63` → `b & kVarintPayloadMask`

* `b & 64` → `b & kVarintContinueMask`

* `>> 6` / `<<= 6` → `>> kVarintPayloadBits` / `<<= kVarintPayloadBits`

从而让编码语义一眼可读。

#### 2.2 `depth_and_lasti` 的位语义显式化

目前：

* `out.stack_depth = depth_and_lasti >> 1;`

* `out.need_push_lasti = (depth_and_lasti & 1) != 0;`

建议用局部 helper 表达意图（同样放匿名命名空间）：

* `constexpr int kLastiFlagMask = 0x1;`

* `constexpr int kStackDepthShift = 1;`

* `int DecodeStackDepth(int depth_and_lasti)`

* `bool DecodeNeedPushLasti(int depth_and_lasti)`

并将变量名从 `depth_and_lasti` 改为 `depth_and_lasti_bits` 或 `packed_depth_and_lasti`，避免读者误解其为普通数值。

（可选）若后续字段增加，可将其升级为 `enum class` 或小型 `struct PackedDepthAndLasti { ... }`，但当前两位语义用 `constexpr + helper` 已足够，不必过度设计。

#### 2.3 “2 倍转换”常量化

把 `* 2` 替换为 `CodeUnitsToBytes(...)` 或 `kCodeUnitSizeInBytes`，并统一在一处做溢出与合法性检查。

### 3) 用测试锁定不变量（避免回归）

目标是用测试把“单位一致性”钉死：

* 新增/扩展解释器端到端用例：覆盖

  * try/except 命中 handler 的基本路径

  * 嵌套 try/except（不同层 handler）

  * raise/re-raise（覆盖 `need_push_lasti` 相关路径）

* 测试应确保：

  * 在特定抛错点落到预期的 handler（可通过打印/断言观察到）

  * 修改 `kCodeUnitSizeInBytes` 或误改 pc 单位后会失败

实现时优先复用现有 `BasicInterpreterTest` 与统一的异常断言 helper（避免在多个用例里重复 pending-exception 样板）。

### 4) 在 `AGENTS.md` 固化“禁止 magic number”的规范

在 **代码规范章节**（建议放在 `4.0 AI 自检清单` 或 `4.2 整体代码风格` 的补充条款）新增明确规则：

* 禁止在实现中直接出现意义不明的数字常量（magic number），尤其是位掩码、移位位数、协议常量、单位换算系数等。

* 必须使用：

  * 匿名命名空间内的 `constexpr` 常量（`kPascalCase`，并限制作用域）

  * 或 `enum class` 表达状态/分类型数字

* 允许的例外：`0/1` 作为数组下标、循环边界、简单布尔转整型等“无歧义语义”场景；即便如此，若牵涉协议/布局/单位，也应命名化。

并给出 1-2 个简短示例（“坏例子/好例子”）让 AI 与人类都能按同一标准执行。

## 交付物

* `src/interpreter/exception-table.cc`：无 magic number；单位转换集中且带防护；变量命名更清晰；必要的中文注释解释 why/how。

* （必要时）`src/interpreter/exception-table.h`：接口注释明确参数单位。

* 单元测试：新增或扩展解释器异常相关用例，锁定查表与 handler 命中行为。

* `AGENTS.md`：新增“禁止 magic number”条款与示例。

## 验收标准（Checklist）

* 查表单位不变量在代码中“显式可见”（函数注释 + helper/类型别名）。

* `exception-table.cc` 中不再出现 2/6/63/64/1 等意义不明数字（全部被命名常量或 helper 吸收）。

* 新增测试能覆盖 handler 命中与 `need_push_lasti` 路径，且能在单位被误改时产生稳定失败。

* `AGENTS.md` 中已加入 magic number 规范，且措辞清晰、可执行。
